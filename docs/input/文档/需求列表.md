## 一、功能需求

### 1. 用户与权限

- **F-001**: 通过学校 CAS 单点登录，全程无自建账号密码。（角色：全部；优先级：P0）
- **F-002**: 登录态在网关统一校验，签发内部短时令牌贯通各微服务。（角色：系统；优先级：P0）
- **F-003**: RBAC 区分五类角色：辅导员 / 学院管理 / 学工管理 / 领导 / 运维（网关 + 方法级双重鉴权）。（角色：系统；优先级：P0）
- **F-004**: 辅导员仅可访问本人责任学生的数据。（角色：辅导员；优先级：P0）
- **F-005**: 运维角色默认不可查看业务明细。（角色：运维；优先级：P0）
- **F-006**: 心理关注标签仅授权角色可见，越权访问留痕。（角色：全部；优先级：P0）
- **F-007**: 账号/角色/组织树/辅导员↔责任学生关系从权威源同步 ⚠️（来源与方式待确认）。（角色：系统；优先级：P0）
- **F-008**: 数据范围可配置（字段级脱敏，如手机号脱敏展示）。（角色：运维/学工；优先级：P1）
- **F-009**: ⚠️ 学生端（知情/查询/申诉）首期不做，后续按需评估。（状态：暂缓）

### 2. 数据接入与质量

#### P0 最小闭环数据源

- **D-001**: 学生基础与辅导员关系 — 核心字段：学号/姓名/学院/专业/年级/培养层次/班级/辅导员映射。（责任部门：教务/学工；优先级：P0）
- **D-002**: 住宿信息 — 核心字段：楼栋/房间/床位/入住状态。（责任部门：后勤；优先级：P0）
- **D-003**: 一卡通消费 — 核心字段：交易时间/金额/商户/消费类型。（责任部门：信息中心/后勤；优先级：P0）
- **D-004**: 校门门禁 — 核心字段：时间/闸口/方向/设备ID。（责任部门：保卫；优先级：P0）
- **D-005**: 宿舍门禁 — 核心字段：时间/楼栋/方向/设备ID。（责任部门：后勤/保卫；优先级：P0）
- **D-006**: 请销假与离返校 — 核心字段：类型/起止时间/事由/审批状态/返校时间。（责任部门：学工；优先级：P0）

#### P1 增强研判数据源

- **D-007**: 学业（挂科/延毕/GPA） — 挂科数/绩点/论文状态/延毕标记。（优先级：P1）
- **D-008**: 重点关爱名单 — 与学号关联，授权可见。（优先级：P1）
- **D-009**: 心理关注脱敏标签 — 仅授权角色可见，不含诊断细节。（优先级：P1）
- **D-010**: 经济困难学生名单 — 与学号关联，最小化可见。（优先级：P1）
- **D-011**: 辅导员进宿舍记录 — 走访时间/楼栋。（优先级：P1）

#### 重点专项

- **D-012**: 夜间上网汇总 — 只接：账号/学号映射、上下线时间、在线时长/会话数、流量/接入区域/终端数 ⚠️（授权需已获批）；禁接：网址/应用名/搜索/聊天/报文内容。（优先级：P1）

#### 数据质量管理（贯穿全部数据源）

- **D-013**: 每个数据源明确责任部门、关联主键、更新频率、质量检查项、授权边界（一数一源）。
- **D-014**: 学号 ↔ 一卡通卡号 / 校园网账号主键映射表建立与维护 ⚠️。
- **D-015**: 数据质量面板：主键完整性 / 连续性 / 覆盖率 / 最近更新时间实时呈现。
- **D-016**: 数据质量不达标自动熔断：相关规则停用、不产正式线索（可手动恢复）。
- **D-017**: 历史数据可回溯 ≥ 56 日（支撑个人基线计算） ⚠️。

### 3. 智能研判引擎（观澜五镜）

- **L1 定矩镜**
  - **R-001**: 规则引擎：阈值 / 持续时间 / 变化幅度 / 组合规则 / 白名单豁免 / 关怀等级映射 / 跟进时限，全部可配置。（建设阶段：生产底座）
  - **R-002**: 场景排除机制：请假 / 实习 / 假期 / 设备故障时不触发或降级。（建设阶段：生产底座）
  - **R-003**: 7 日内 2 类独立线索叠加自动提升核实优先级。（建设阶段：生产底座）

- **L2 观变镜**
  - **R-004**: 个人基线（28/56 日历史均值 + 分位数）计算，检测变化幅度与异常拐点。（优先建设）
  - **R-005**: 个人基线异常检测（非全校统一阈值）。（优先建设）

- **L3 合证镜**
  - **R-006**: 多线索合证 Top-k 排序（关怀等级 + 独立线索数 + 时效 + 数据可信度）。（灰度验证）

- **L4 时序镜**
  - **R-007**: 时间链路 / 趋势演化 / 群体共振分析 ⚠️。（研究储备）

- **贯穿 护航镜**
  - **R-008**: 所有线索必须附可读证据链 + 解释文本；辅导员为最终核实人；不得自动作出重大不利决定。（贯穿）

> ⚠️ AI 范围待拍板（PRD D 组）：首期生产范围是“仅规则引擎”、“规则+个人基线”还是“全量分期含大模型”，直接影响排期。ROADMAP 暂定按“规则引擎 + 基线”设计主干。

### 4. 线索生成与证据

- **C-001**: 线索含字段：学号 / 触发规则 / 类别（学业/经济/安全）/ 关怀等级（Ⅰ/Ⅱ/Ⅲ）/ 评分 / 独立线索数 / 数据可信度 / 生成时间 / 跟进截止时间 / 状态 / 解释文本 / 关怀建议。
- **C-002**: 每条线索附证据：数据源 / 指标项 / 当前值 / 基线值 / 变化幅度 / 时间窗 / 排除项命中情况。
- **C-003**: 关怀等级 Ⅰ 提醒（时限 ⚠️）/ Ⅱ 重点（48h 内）/ Ⅲ 重点关怀（24h 内）。
- **C-004**: 线索状态流转：待核实 → 处理中 → 已跟进（或超期 / 已关闭 / 已熔断）。
- **C-005**: 白名单命中时不产线索，可审计。

### 5. 四个原型场景

- **S-001 毕业季关怀**：挂科/延毕信号 + 近期消费基线下降 + 深夜连续活跃 → 生成关怀线索，建议联动导师/教务/心理，设 7 天持续关注。
- **S-002 隐性经济压力**：连续低消费（对比个人历史基线，非全校均值）+ 困难身份（授权）+ 无请假 → 排除节食/校外就餐/借卡/漏数后介入；最小范围可见，不贴“贫困”标签。
- **S-003 住宿安全**：连续多日无校门/宿舍记录 → 多源排除（请假/实习/无门禁覆盖/设备故障）→ 中关怀线索（48h 核实）；门禁数据中断则自动熔断。
- **S-004 夜间作息**：校园网数据只回答“作息是否变化”；升级条件 = 连续异常夜 + 较个人基线显著上升 + 独立印证线索；禁止展示用途内容，禁止心理/安全自动诊断。

### 6. 辅导员工作台

- **W-001**: 工作台统计：待核实数 / 处理中 / 今日已跟进 / 学院超期统计 / 今日核实容量。
- **W-002**: Top-k 优先级列表（关怀等级 + 独立线索数 + 时效 + 数据可信度排序）。
- **W-003**: 线索详情页：学生信息 + 证据链 + 解释文本 + 关怀建议。
- **W-004**: 核实反馈录入字段：联系情况 / 核实结果（属实/噪声/待观察）/ 噪声原因 / 关怀方式 / 是否转介 / 持续关注设置 / 备注。
- **W-005**: 每步反馈有责任人、时限、留痕；超期状态自动标记。

### 7. 学生详情（一人一档）

- **P-001**: 学生基础信息（学号/姓名/学院/专业/年级/培养层次/班级/学籍状态/辅导员）。
- **P-002**: 动态信号聚合：学业 / 经济 / 安全三类历史信号时间线。
- **P-003**: 历史线索列表与跟进记录。
- **P-004**: 关爱标签（授权范围内可见）展示。

### 8. 协同转介

- **T-001**: 发起转介到：心理 / 资助 / 教务 / 保卫 / 学院 / 家长 ⚠️（渠道与合规边界待确认）。
- **T-002**: 转介状态流转与处理结果回填。
- **T-003**: 心理相关转介仅授权角色可发起和接收。
- **T-004**: 跨部门协同是平台内流转还是需对接外部系统 ⚠️。

### 9. 规则配置

- **RC-001**: 规则配置项：阈值 / 持续时间 / 变化幅度 / 组合条件 / 白名单/豁免 / 关怀等级映射 / 跟进时限。
- **RC-002**: 规则版本管理（草稿/灰度/生产/熔断/回退），修改需审计。
- **RC-003**: ⚠️ 规则权限是否下放学院待确认，变更审批流程待定。
- **RC-004**: 知识库/关怀建议/谈心谈话建议维护（模板库还是模型生成 ⚠️）。

### 10. 学院管理端

- **I-001**: 本学院范围线索与闭环情况查看。
- **I-002**: 超期督办与跟踪提醒。
- **I-003**: 院内协同发起。

### 11. 领导驾驶舱

- **L-001**: 指标：今日新增线索 / 重点关怀学生数 / 闭环率 / 噪声率 / 近 7 日趋势 / 各学院关怀与跟进对比。
- **L-002**: 默认汇总展示，下钻学生明细需授权 + 业务必要 + 留痕。
- **L-003**: 跨学院态势对比。

### 12. 周报/月报与运营复盘

- **OP-001**: 自动生成周/月报：闭环率 / 噪声率 / 噪声原因 / 协同成效统计。
- **OP-002**: 按学院维度对比。
- **OP-003**: 规则校准建议（基于复盘数据）。

### 13. 学生标签体系

- **ST-001**: 标签类型：困难身份 / 心理关注脱敏标签 / 重点关爱名单 / 经济困难。
- **ST-002**: 每类标签明确来源 / 授权可见角色 / 生效时间。
- **ST-003**: 心理标签最小范围可见；不与“贫困”等负面标签直接展示。
- **ST-004**: ⚠️ 标签来自源系统同步还是平台内人工维护待确认。

### 14. AI 解释与关怀建议（分期）

- **AI-001**: 每条线索生成可解释依据（可解释、可追溯、可拒绝、可回退）。（阶段：L2 灰度）
- **AI-002**: 关怀路径推荐。（阶段：L3）
- **AI-003**: 谈心谈话建议生成。（阶段：L3）
- **AI-004**: 自然语言问数（统计/态势查询） ⚠️。（阶段：L3）
- **AI-005**: ⚠️ 若用外部大模型 API，须先过 PIPL 敏感数据外发评估；原则上优先自建/本地化。

### 15. 模型与质量治理（分期）

- **MG-001**: 候选模型上线须通过四关卡：数据质量 → 效果评估（Precision@K、Recall@K、噪声率）→ 分群公平性（本科/研/实习） → 合规人工复核。
- **MG-002**: 灰度发布：任一关卡未通过停止扩大范围。
- **MG-003**: 稳定版本保留 + 一键回退；质量故障或效果下降立即回退规则基线。
- **MG-004**: 效果评估不以单一 AUC/总体准确率为上线依据。

### 16. 审计与留痕（贯穿全部模块）

- **AU-001**: 审计覆盖：查看 / 导出 / 跟进 / 规则修改 / 白名单调整 / 越权尝试。
- **AU-002**: 每条审计记录包含：操作人 / 角色 / 操作类型 / 对象 / 时间 / 来源 IP / traceId。
- **AU-003**: 审计日志留存 ≥ 6 个月，防篡改。
- **AU-004**: 全网 NTP 校时，确保时间一致性。

## 二、非功能需求

### 性能

- **NF-001**: 峰值并发用户（辅导员/管理者为主）：⚠️ 建议 500–1000，待确认。
- **NF-002**: 工作台列表/详情 P95 响应时延：⚠️ ≤ 2s，待确认。
- **NF-003**: 研判作业时效（个人基线/Top-k）：⚠️ 离线/准实时批处理，每日 ≥ 1 次；安全类近实时，待确认。
- **NF-004**: 日增事件数据量（消费/门禁）：⚠️ 约数百万条/日（5 万人），待确认。

### 可用性

- **NF-005**: 业务时段可用性：⚠️ ≥ 99.9%，安全关怀链路更高，待确认。
- **NF-006**: 灾备 RPO/RTO：⚠️ 待确认。

### 安全与合规

- **SEC-001**: 目标合规：等保三级 + PIPL。
- **SEC-002**: 全链路 HTTPS（必要时国密 SM2/SM3/SM4）。
- **SEC-003**: 敏感字段（手机号/心理标签等）加密存储（SM4/AES）；数据库 TDE；密钥集中管理。
- **SEC-004**: 参数化查询（PreparedStatement / JPA / MyBatis #{}），禁止字符串拼接 SQL。
- **SEC-005**: 所有外部输入做服务端校验（Bean Validation），输出按上下文转义，防注入/XSS。
- **SEC-006**: 密钥/口令走环境变量或 Nacos 配置中心，禁止硬编码/入库。
- **SEC-007**: 安全测试（渗透/漏扫）通过；WAF / 堡垒机 / 安全域（DMZ/应用区/数据区）到位。
- **SEC-008**: ⚠️ 数据保留与删除策略（毕业/离校后处理）待确认。
- **SEC-009**: ⚠️ 处理学生个人信息的法律依据（知情同意 vs 履行法定职责）及告知方案待确认。

### 可维护性

- **MN-001**: 全链路 traceId，日志带 traceId 与 MDC。
- **MN-002**: Prometheus + Grafana 监控；SkyWalking 链路追踪（分期接入）。
- **MN-003**: 规则可配置、可解释、可熔断、可版本回退。
- **MN-004**: 数据质量监控面板。
- **MN-005**: 灰度发布与版本管理。

## 三、集成需求

- **IN-001**: 学校统一身份认证 CAS — 协议版本/返回属性/登出策略 ⚠️ 待确认。
- **IN-002**: 统一门户（微前端嵌入） — qiankun / 无界，按门户实际方案 ⚠️。
- **IN-003**: Nacos（注册+配置中心） — Spring Cloud Alibaba。
- **IN-004**: 各数据源部门 — P0/P1 对接方式（API/ETL/批）⚠️ 待逐一确认。
- **IN-005**: 通知渠道 — 站内消息为基础；⚠️短信/企业微信/校园 App 待确认。
- **IN-006**: 心理/资助/教务/保卫外部系统 — ⚠️ 平台内流转还是外部系统对接待确认。

## 四、里程碑与交付顺序

- **M0 脚手架**: 骨架/CI/基础设施/安全基线 — ✅ 已完成。
- **M1 CAS SSO + 权限骨架**: 统一身份认证 + RBAC + 数据范围 + 登录审计 — ⏭️ 下一步（铁律一：先计划）。
- **M2 数据接入与质量底座**: P0 数据接入 / 主键映射 / 质量监控 / 熔断 — 待办。
- **M3 规则引擎 + 单场景闭环**: 定矩镜 L1 + 第一个场景端到端闭环 — 待办。
- **M4 多场景工作台 + Top-k + 协同转介** — 待办。
- **M5 学院端 + 领导驾驶舱 + 周报月报** — 待办。
- **M6 安全合规加固**: 等保三级/PIPL 测评就绪 — 待办。
- **M7 学院试运行与常态化运营**: 1–2 学院试运行，噪声率/闭环率验收 — 待办。
- **（可选）ML 增量**: 个人基线（L2）灰度 → AI 解释/自然语言问数（L3）→ 模型治理四关卡 — 待 D 组确认。
