# “学林知微”方案、UX 与品牌输入对账报告

> 对账输入：`extract-solution-ux.md`（覆盖方案 PPT、Web UX PDF、移动 UX PDF、苏大校徽）  
> 对照基线：`prd.md`、`addendum.md`  
> 对账日期：2026-07-16  
> 约束：本报告只识别输入吸收情况与差距，不修改 PRD。

## 1. 结论

当前 PRD 已准确吸收“数智学工重要模块”“线索而非结论”“人工核实闭环”“Top-k 减负”“角色化首页”“Web 管理 + 移动轻处置”等核心产品要求，主体功能覆盖度较高。

仍有两类高影响遗漏：一是方案中的“观澜智核/观澜五镜”能力品牌及其分层建设逻辑未进入 PRD 或补充材料，导致方案叙事与正式需求之间失去可识别映射；二是学校 Web、移动与校徽规范只被概括为“遵循既定 UI 规范”，尚未形成足以约束 UX、验收和品牌层级的明确要求。此外，七步闭环虽在功能上基本覆盖，但没有按来源的七阶段形成统一过程模型与状态映射，可能造成下游对“线索推送”和“规则优化”边界理解不一致。

## 3. 最重要差距（按影响排序）

### Gap 1｜“观澜智核/观澜五镜”整体缺失

**严重度：高。**

PRD 已把定矩、观变、合证、时序与护航对应的功能拆入规则、个人基线、Top-k、后续增强、质量与回退要求，但没有保留“观澜智核”品牌，也没有建立五镜与 FR、分期之间的映射。这不是单纯文案遗漏：它会造成正式 PRD 与解决方案核心能力架构失去共同语言，影响后续架构分层、UX 命名、汇报沟通和范围验收。

建议在 PRD 产品定位或能力概览中增加一段非技术性能力映射，并明确：L1 定矩镜、L2 观变镜为首期生产核心；L3 合证镜用于可解释 Top-k；L4 时序镜属于后续增强；护航镜为证据、质量、人工核实和回退的横贯能力。“观澜智核”作为智能能力品牌，层级低于“学林知微”。

### Gap 2｜学校品牌与 UI 规范未形成强制、可验收约束

**严重度：高。**

PRD 目前只有“适配学校既定 UI 规范”的原则性表述，未保留 Web `#AF251B`、移动 `#D03D37` 的端别差异，也未明确 1440×900 / 375 px 基准、统一门户壳层和既有组件复用。来源明确要求学校 UI 规范为强约束，PPT 蓝绿视觉不得覆盖学校红色品牌体系。若不补入，UX 和研发仍可能在不违反现有 PRD 字面的情况下另造组件或沿用 PPT 风格。

建议在 NFR 或独立“设计体系与品牌约束”中定义：端别规范优先级、统一壳层复用、组件复用、响应基准、状态不可仅靠颜色表达，以及禁止以 PPT 配色替代学校品牌色。

### Gap 3｜校徽资产与品牌层级完全未定义

**严重度：中高。**

输入明确给出苏大校徽资产及不可拉伸、裁切、重绘、改色、叠加效果、重复堆叠等要求，并规定学校/数智学工主品牌高于“学林知微/观澜智核”。PRD 与 addendum 均未承接。该遗漏会直接影响登录、页头、导出报告、驾驶舱大屏和宣传截图的一致性。

建议由 PRD 固化品牌层级和资产使用原则，具体安全留白、尺寸与落位交给 UX 规格承接。

### Gap 4｜七步闭环存在功能等价，但缺少统一阶段模型

**严重度：中。**

来源的七步为“数据归集—智能研判—线索推送—辅导员初审/核实—关爱干预—结果反馈—规则优化”；PRD 改写成“数据可用—线索生成—人工核实—关怀跟进—协同转介—结果回填—规则复盘”。能力并未丢失，但“线索推送”从过程阶段变为消息/待办能力，“协同转介”则被提升为一个阶段。

建议保留来源七步作为唯一业务闭环骨架，将协同转介定义为“关爱干预”中的可选分支，并把每一阶段映射到责任角色、关键状态和 FR。这样可避免运营指标、流程图与状态机使用不同口径。

### Gap 5｜学生价值与“温度”叙事被功能治理语义稀释

**严重度：中。**

PRD 对辅导员效率、学院治理和校级态势表达充分，但对学生侧价值——更早被关心、避免标签化、获得适切支持——主要散落在愿景、旅程和非目标中。来源强调“AI 辅助研判 × 人工审核保障温度”与“更早洞察、更暖抵达、更清治理”，这些定性价值值得在产品价值或成功原则中明确保留。

建议增加简短的分角色价值表述，不新增学生端范围，也不把情感价值误写成不可验收的口号。

## 4. 已充分吸收、无需重复扩写的内容

- 项目归属于数智学工大系统，并复用统一门户、身份、消息和待办能力。
- “线索而非结论”“辅助而非替代”、重大行动必须人工决定。
- 基于证据链、个人基线、场景排除、数据可信度进行解释和研判。
- Top-k 行动队列、SLA、责任人、状态、核实反馈、持续观察、转介与复盘。
- 领导看汇总态势，辅导员和授权执行者查看必要明细。
- Web 用于高密度管理分析，移动用于提醒、摘要和轻量处置，且两端共享唯一状态。
- 避免诊断、污名化和恐慌性表达；颜色不是等级或错误的唯一表达方式。
- 审批、数据与合规事项均为已解决前提，不作为开放问题或阻塞项。

## 5. 推荐修订落点（供主流程裁决）

1. 在“产品定位与边界”增加“品牌与能力体系”：学林知微、观澜智核、观澜五镜的关系及分期映射。
2. 在“产品愿景/原则”补足学生价值与“人工审核保障温度”的定性语义。
3. 在功能需求前增加七步业务闭环简图或表格，并映射现有 FR，不重写功能章节。
4. 将 §9.5 扩展为可验收的“体验一致性、设计体系与品牌”要求，明确 Web、移动端别规范优先级及组件复用。
5. 将校徽使用、模块品牌层级和 PPT 视觉非产品规范写入 PRD 原则；具体 token、尺寸和安全留白交由后续 UX 规格承接。
