## 2. 产品方案核心提取

### 2.1 产品名称与定位

- 中文名：**学林知微**。
- 英文名：**ScholarSense**。
- 智能能力品牌：**观澜智核**。
- 归属：数智学工平台的**学生成长关怀模块**，不是独立于数智学工之外的新平台。
- 方案中的标准表述：学生成长洞察与关爱协同；以数据发现细微信号，以 AI 辅助研判，以人工审核保障温度。
- 产品定位公式：**数据发现细微信号 × AI 辅助研判 × 人工审核保障温度**。
- 目标不是捕捉或判定“异常”，而是让有限的学生关爱资源更早、更准地抵达需要帮助的学生。

### 2.2 需要解决的核心问题

方案将当前问题归纳为四类：

1. **发现滞后**：问题常在请假、成绩或危机显性化后才被注意。
2. **数据分散**：消费、门禁、住宿、学业等信息分布在不同系统或部门。
3. **排查压力**：辅导员依赖经验人工筛查大量学生，无法持续聚焦最需要优先核实的人。
4. **闭环不足**：名单或线索推送后，责任人、时限、反馈、协同和规则复盘缺少统一承载。

正式 PRD 应把问题重心写成“发现—核实—关爱—反馈—优化”的业务机制缺口，而不是单纯建设一个风险名单页面或算法模型。

### 2.3 价值叙事

#### 学生价值

- 学生状态变化能够更早被关心和看见。
- 关爱更及时、更有尊严，避免简单标签化。
- 通过人工理解真实情境，获得谈心、资助、学业、心理、住宿安全等适切支持。

#### 辅导员价值

- 从大范围人工排查转为聚焦最值得优先核实的 Top-k 学生。
- 获得结构化证据链、排除项、解释和关怀路径建议。
- 用统一工作台承载核实、跟进、协同、持续关注和结果反馈。

#### 学院价值

- 掌握本学院关怀态势、重点对象、超期事项和闭环成效。
- 建立辅导员、学院与相关支持部门之间的协同抓手。
- 通过噪声原因与反馈校准规则，提升治理质量。

#### 学校价值

- 形成全校学生成长关怀态势与趋势研判能力。
- 从数据到行动的过程可追踪、结果可复盘、机制可持续。
- 将跨部门数据能力转化为数智学工中的常态化关爱运营能力。

价值主线可压缩为：**更早洞察、更暖抵达、更清治理**。

## 3. 产品原则与语义边界

方案给出六项原则，正式 PRD 宜作为产品决策原则保留：

1. 关爱优先：服务学生成长，不做简单管控。
2. 规则可解释：每条线索说明触发事实、时间和排除结果。
3. AI 辅助：AI 生成解释、排序和建议，不替代辅导员判断。
4. 分级关怀：根据证据强度、多线索印证与时效分层跟进。
5. 闭环干预：不止推送名单，必须记录责任、时限、过程和结果。
6. 安全可控：按角色展示必要信息，关键操作留痕，数据故障时熔断或回退。

三条强产品语义边界：

- **线索，而非结论。**
- **建议，而非自动决策。**
- **按角色与业务需要呈现，而非所有用户查看全部明细。**

用户已明确所有合规事项均解决，因此这些内容在 PRD 中应转写为已确定的产品行为、权限规则和质量要求，不应出现“待审批”“待授权”“待评审”“合规风险”等开放表述。

## 4. 业务闭环

方案定义完整七步闭环：

1. 数据归集。
2. 智能研判。
3. 线索推送。
4. 辅导员初审/核实。
5. 关爱干预。
6. 结果反馈。
7. 规则优化。

配套运营机制：周报、月报、噪声原因分析、协同成效分析。

每一步应具有：

- 明确责任角色；
- 状态与状态转换；
- 处理时限；
- 支撑证据；
- 操作与反馈记录；
- 异常、退回、转交、持续关注与关闭路径。

PRD 不应把“生成线索”视为完成，而应把“线索被核实、采取行动、形成结果反馈并回流优化”定义为核心闭环。

## 5. 模块边界与能力架构

### 5.1 四类服务入口

1. **领导驾驶舱**：看汇总态势、趋势、闭环率、噪声率、学院跟进和协同提醒；默认不下钻不必要的学生细节。
2. **学工管理端**：进行全校业务运营、规则与数据质量管理、跨学院协同、周报月报等。
3. **学院管理端**：查看学院范围内关怀态势、线索分布、辅导员处理效能、超期与协同事项。
4. **辅导员工作台**：处理责任学生的待核实线索、跟进任务、持续关注和反馈。

这四个入口应作为角色视图或角色化工作空间纳入数智学工统一门户，不宜被实现为四套割裂产品。

### 5.2 业务应用层

- 成长总览。
- 线索列表。
- 学生详情/一人一档。
- 协同闭环。
- 规则配置。
- 数据质量。
- 周报/月报与运营复盘。

### 5.3 观澜智核能力层

方案提出“观澜五镜”：

- **定矩镜（L1）**：制度阈值、白名单/豁免、组合规则、跟进时限；生产底座。
- **观变镜（L2）**：个人基线、变化检测、分位漂移、异常拐点；优先建设。
- **合证镜（L3）**：多线索合证、Top-k 排序，让有限资源优先抵达；灰度验证。
- **时序镜（L4）**：时间链路、关系图谱、趋势演化、群体共振；研究储备。
- **护航镜**：证据链、人工核实、质量控制、回退等横贯能力。

此外包括：学生标签体系、规则引擎、AI 解释、关怀建议、知识库、自然语言问数、周报生成与校准建议。

范围建议：首期 PRD 应明确 L1 定矩镜与 L2 观变镜为生产核心；L3 可纳入受控排序能力；L4 时序镜按方案自身定位应列为后续能力，不进入首期强承诺，除非需求清单另有明确要求。

### 5.4 数据底座

来源列举的数据域：

- 学生基础信息与学生—辅导员责任关系；
- 住宿信息；
- 一卡通消费；
- 校门门禁与宿舍门禁；
- 学业信息；
- 请销假、离返校；
- 关爱标签；
- 夜间上网汇总；
- 辅导员进宿舍/走访等人工业务记录。

用户已明确数据问题已解决，PRD 应将上述数据域视为已具备来源与接入条件；仍需保留一数一源、关联主键、更新频率、质量校验、数据中断熔断等工程性要求。

## 6. 关怀场景与研判逻辑

### 6.1 三大类线索

#### 学业关怀

- 挂科与绩点下降；
- 缺勤/学习参与下降；
- 论文未通过；
- 毕业与延毕关怀信号。

#### 经济关怀

- 连续无消费；
- 餐均消费偏低；
- 近期消费较个人基线显著下降；
- 困难身份与行为变化叠加。

#### 安全关怀

- 晚归与夜间出入；
- 连续无门禁/无校园行为；
- 疑似校外住宿；
- 离校未返/逾期未销假。

方案还把夜间作息变化作为专项信号，通过夜间在线时段、时长、会话数、流量汇总、接入区域、终端数和个人基线进行研判。

### 6.2 研判四步

1. 行为证据：阈值、持续时间、变化幅度。
2. 场景排除：请假、实习、假期、住宿变更、设备或接口故障等。
3. 独立印证：消费、门禁、学业、走访等跨域信号相互印证。
4. 关注优先：授权标签只提升核实优先级，不直接定性。

方案提出“任意 2 类独立线索在 7 日内叠加，可提升核实优先级”，可作为组合规则示例，正式 PRD 应避免把示例固化为唯一规则；阈值和窗口宜配置化。

### 6.3 分级跟进

- I 级｜提醒：谈心谈话、关心了解。来源未给定明确 SLA。
- II 级｜重点：介入干预、持续跟踪，48 小时内跟进。
- III 级｜重点关怀：紧急核实、多方联动，24 小时内跟进。

正式 PRD 需要统一等级命名、展示样式、SLA 起算点、暂停/延期规则、升级降级条件和超期计算方式。

### 6.4 四个代表性故事

1. **毕业季压力**：毕业年级 + 学业/延毕信号 + 消费下降 + 连续夜间作息变化；建议核实论文、答辩、就业、家庭压力，并可联动导师、教务或心理支持。
2. **隐性经济压力**：低消费必须结合个人历史基线、身份与人工核实；先排除节食、校外就餐、借卡、漏数，再匹配资助支持；不贴“贫困”结论标签。
3. **住宿安全**：连续无门禁需排除请假、实习、校外住宿、无设备覆盖和接口故障；必要时联动学院、家长或保卫；门禁数据中断则相关规则熔断。
4. **夜间作息变化**：仅回答作息是否显著变化；结合个人 28/56 日基线、连续异常夜和其他独立线索，不以单夜高流量直接定性。

## 7. 核心页面与信息架构

### 7.1 建议一级导航

正式 PRD 可采用以下稳定信息架构，并嵌入数智学工统一导航：

1. 工作台/成长总览。
2. 关怀线索。
3. 学生档案。
4. 关怀协同。
5. 运营分析。
6. 规则与模型。
7. 数据质量。
8. 系统配置（按角色可见）。

其中“领导驾驶舱”更适合作为领导角色的默认首页，而不是面向所有用户的一级菜单。

### 7.2 辅导员工作台

方案明确的关键区域：

- 我的待核实数量；
- 处理中数量；
- 今日已跟进数量；
- 学院超期数量；
- 今日核实容量（已安排/容量）；
- 按关怀等级、独立线索数、时效和数据可信度排序的 Top-k 列表；
- 单条线索呈现学生、主题、得分/优先级、线索数量、SLA 与处理入口。

结构化反馈字段：联系情况、核实结果、关怀方式、噪声原因、是否持续关注、是否转交协同。

体验重点：用户进入后应立刻知道“今天先处理谁、为什么、何时到期、下一步做什么”。

### 7.3 领导驾驶舱

方案中的示例指标：

- 今日新增线索；
- 今日重点关怀学生；
- 本周闭环率；
- 近 30 日噪声率；
- 最近 7 日关怀趋势；
- 各学院线索量与跟进/闭环情况。

体验边界：以校级/学院汇总、趋势、对比、超期和协同提醒为主；下钻遵循角色和业务需要，避免把敏感学生明细作为领导首页常驻信息。

### 7.4 线索详情/学生详情

从方案故事可推导的核心信息块：

- 学生基础与责任关系；
- 当前关怀等级、主题、状态、责任人和 SLA；
- 触发事实、时间轴、个人基线与变化幅度；
- 多线索证据链；
- 已完成的排除项及数据可信度；
- AI 解释与关怀建议（显式标注为建议）；
- 历史线索、跟进记录和持续关注计划；
- 转介/协同记录；
- 快捷处理动作与完整操作日志。

### 7.5 规则与数据质量

- 规则配置：阈值、时间窗口、组合条件、适用人群、豁免、等级、SLA、启停与版本。
- 质量管理：主键、连续性、覆盖率、更新时间、异常状态、受影响规则、熔断/恢复状态。
- 模型/排序评估：Precision@K、Recall@K、噪声率、闭环/协同成效、分群差异、版本与回退。

方案明确模型效果应围绕有限跟进能力评估，不宜仅展示 AUC 或总体准确率。

## 8. 用户体验原则

1. **行动优先**：页面首先回答待办、优先级、时限和下一步，不用海量图表淹没行动。
2. **解释就地呈现**：优先级、等级、AI 建议旁直接展示关键触发事实和排除项。
3. **人工掌舵**：提供采纳、修改、拒绝、标记噪声、转交、持续关注等动作；不把建议做成不可编辑结论。
4. **闭环可见**：列表与详情均展示状态、责任人、剩余时限和最近处理进展。
5. **分层披露**：摘要—证据—原始趋势渐进展开；领导看汇总，执行人员看必要明细。
6. **避免污名化语言**：使用“关怀线索、待核实、变化、建议关注”等语汇，避免“问题学生、异常人员、判定”等标签。
7. **错误与空状态可行动**：明确无数据、数据延迟、规则熔断、无权限、加载失败等状态，并给出刷新、查看原因或联系责任方的路径。
8. **规则与模型透明**：用户可看到版本、生成时间、有效期、适用范围和数据可信度。
9. **跨端一致、任务有别**：桌面端承担高信息密度管理与分析；移动端适合提醒、轻量核实、快速跟进和查看摘要，不应机械缩放桌面驾驶舱。

## 9. Web 端视觉与组件约束

来源：《校园服务总入口 WEB 端设计规范》。该规范基于 1440×900 px，应作为数智学工统一 Web 门户的视觉基线。

### 9.1 色彩

#### 品牌主色

- 默认：`#AF251B`
- 悬浮：`#C53227`
- 点击：`#A7180D`

#### 文字

- 强调/正文标题：`#1A1C1C`
- 次强调/正文标题：`#333333`
- 次要信息：`#666666`
- 置灰信息：`#999999`
- 禁用：`#AAAAAA`
- 纯白文字：`#FFFFFF`
- 有底色标题：`#594343`
- 有底色按钮文字：`#464359`

#### 线条与填充

- 按钮描边：`#E8ECF0`
- 分割线：`#EAEAEA`
- 纯白填充：`#FFFFFF`
- 禁用填充：`#ECECEC`
- 常规浅底：`#FBFBFC`
- 状态底色：`#FFD8D8`、`#D2EFD1`、`#F8E6D0`、`#EBEBEB`

#### 功能辅助色

- 失败/警示/删除：`#FF3B3B`
- 成功/完成/编辑：`#1DAF1B`
- 提示/状态：`#DE8416`
- 导出/按钮/状态：`#247FFF`

PRD 可写语义色要求，不应在每个需求中重复硬编码色值；色值应进入设计 token 或统一组件库。

### 9.2 字体与字号

- 中文：苹方；英文：SF Pro；数字：苹方（来源表述）。
- 10/12 px：辅助与次要文案。
- 14 px：常规正文。
- 16 px：大正文。
- 18/20/22 px：常规/中/大标题。
- 24 px：特殊标题。
- 字重涵盖 Light、Regular、Middle、Bold；正文以 Light/Regular，标题与强调以 Middle/Bold。

实现时需要为 Windows 和非 Apple 环境提供系统字体回退栈，保证学校终端显示一致。

### 9.3 布局和壳层

- 沿用统一门户页头、登录/用户区、个人数字空间、待办事项入口、页尾与版权信息。
- 使用统一面包屑、单层/多层侧边导航、页签与分页器。
- 阴影示例：`0 5px 14px rgba(182,192,206,.15)`、`5px 5px 20px rgba(182,192,206,.4)`。
- 应避免在模块内创建与统一门户竞争的第二套顶部导航或品牌壳层。

### 9.4 必备组件覆盖

规范已提供或示意：

- 主按钮（填充/线框/文本，大/常规/小，默认/悬浮/点击，半圆角/圆角）；
- 空状态与 404、无网络、无权限、加载中；
- 侧边导航、页签、分页、气泡提示、引导；
- 表格、描述列表；
- 文本/数字输入、选择器、级联、树选择；
- 日期与时间选择；
- 表单与校验；
- 文件/图片上传、预览、下载、删除；
- 单选、开关等选择控件。

学林知微应复用这些统一组件，避免为相同交互另造视觉变体。关怀等级、状态、SLA、数据可信度等可在统一标签/状态组件上扩展语义，而不是使用完全独立的组件体系。

## 10. 移动端视觉与布局约束

来源：《云中苏大 UI 视觉规范设计》，适用于 V1.0，基于 375 px 宽设计。

### 10.1 色彩

- 主色：`#D03D37`，用于强指示控件、标签、首页大板块标题与提交按钮。
- 重要文字：`#333333`。
- 重要/正文辅助文字：`#666666`。
- 次要文字：`#999999` 或 `#9B9B9B`。
- 未选中/弱化文字：`#ACACAC`。
- 分割线：`#ECECEC`。
- 模块背景：`#F7F8FA`。
- 内容背景：`#FFFFFF`。
- 辅助醒目色：`#FC6864`、`#FEAE4A`、`#478FE4`（来源页面中的色名排版存在错位，应以色值为准）。

Web 与移动主色不同：Web 为 `#AF251B`，移动为 `#D03D37`。这是来源明确的端别差异，不能擅自统一为 PPT 的蓝绿色。

### 10.2 字体与字号

- 使用系统原生字体，优先苹方、思源黑体等。
- 16 px：导航栏标题、按钮文字、重要标题、部分正文。
- 15/14/13 px：正文、卡片、输入框、Tab 等常用信息。
- 12 px：辅助说明。
- 10 px：底部导航文字。

### 10.3 图标与控件

- 底部导航常规图标：24×24 px。
- 底部导航中部大图：48×48 px。
- 消息提醒：12×12 px。
- 应用服务大图标：36×36 px；小图标：18×18 px 或特定卡片中 15×15 px。
- 首页金刚位：40×40 px。
- 搜索应用卡片：22×22 px。
- 返回/更多图标：7×14 px。
- 搜索框图标：20×20 px。
- 新上线/建设中/维护中标签：38×13 px。
- 删除/添加应用：40×40 px。
- 缺省图：179×119 px。

### 10.4 布局

- 普通页面两侧留白采用 10 px 或 24 px，根据内容层级与稀疏程度选择；同一层级左右保持一致。
- 首页图标等分布局示例：两侧约 26 px，项目间约 45 px。
- 模块间距来源示例覆盖 6/8/10/12/13/20 px，应在设计阶段收敛为统一间距 token。
- 底部导航存在四项加中部突出入口的既有样式；若学林知微仅作为云中苏大/数智学工内的业务模块，不应新增独立底部导航。

### 10.5 移动端适用边界

方案 PPT 的主要交互是管理端和驾驶舱，移动 UI 规范表明学校具备统一移动入口。PRD 可支持：

- 待办提醒；
- 线索摘要与证据摘要；
- 快速联系/记录核实；
- 选择关怀方式、结果与持续关注；
- 协同转交和查看处理进度。

规则配置、批量运营、复杂趋势分析、数据质量诊断等宜保留在 Web 端。

## 11. 品牌资产约束

- 提供的 `苏大logo.png` 是圆形苏州大学校徽，透明底，321×319 px。
- 不拉伸、不裁切、不重绘、不改变内部颜色，不叠加阴影或渐变。
- 应保持近似 1:1 比例并预留安全留白。
- 用于统一门户壳层、登录/页头或正式导出物时，应遵循数智学工/云中苏大的既有位置和尺寸，不在学林知微模块内重复堆叠校徽。
- “学林知微 ScholarSense / 观澜智核”可以作为模块名与能力品牌，但层级必须低于学校及数智学工主品牌。

## 12. 来源间冲突、过时项与 PRD 归一化建议

### 12.1 已被用户前提覆盖的过时内容

方案第 18、19、20、22 页包含“待整改、待评审、合规审批、授权首批数据、建议领导协调”等表述。根据用户明确指令：

- 不得在 PRD 中保留为前置条件、待办、风险或依赖。
- 可将已经确定的权限、审计、必要字段、人工核实、回退等转写成产品和技术验收要求。
- 数据域应按已具备接入条件处理；数据质量监控仍然是运行期能力，不等于数据接入未解决。

### 12.2 建设阶段表述与正式项目语境冲突

方案仍以“选择 1-2 个学院试运行、协调首期数据、8-12 个月纳入数智学工”为建议，而用户已明确这是某 211 头部高校数智学工的重要模块。正式 PRD 应：

- 明确系统级生产建设目标和校级架构归属；
- 如需分阶段上线，写成发布策略、灰度范围与能力分批交付，而不是项目是否建设、数据是否授权的探索性试点；
- 不应把“纳入数智学工”列为未来事项，因为产品当前即属于数智学工。

### 12.3 名称与等级需统一

- “风险预警/关怀线索/异常/重点关怀”等术语在不同材料中可能混用。方案本身倡导关爱语义，建议正式产品前台统一使用“关怀线索、待核实、关怀等级”。
- “I 级提醒、II 级重点、III 级重点关怀”容易与严重度和学生标签混淆；PRD 应建立唯一术语表和颜色/图标映射。
- “AI 评分/92 分”在辅导员示例中出现，但评分含义未定义。正式需求应明确它是排序分、综合优先级还是置信度，避免用户误读为学生风险概率。

### 12.4 视觉冲突

- 方案 PPT 高频使用深蓝、青绿、蓝灰和橙色，实际字体为 Hiragino Sans GB/Microsoft YaHei，属于演示稿视觉。
- 学校 Web 规范主色为深红 `#AF251B`，移动规范主色为红 `#D03D37`。
- 正式产品必须优先遵循学校统一 Web/移动规范；蓝绿科技感可谨慎用于数据可视化序列，但不能替代品牌主色或壳层。

### 12.5 指标口径待在 PRD 内定义，而非业务阻塞

方案展示闭环率、噪声率、Precision@K、Recall@K、协同成效等指标，但未完整定义分子、分母、时间窗口与重复线索处理。PRD 应给出指标定义或要求后续数据字典承接，不能直接把示例数字当验收目标。

## 13. 建议写入正式 PRD 的关键决策

1. 学林知微是数智学工中的学生成长关怀模块，统一门户、统一身份、统一设计体系。
2. 核心业务对象是“待人工核实的关怀线索”，不是自动生成的学生结论。
3. MVP/首期必须覆盖完整闭环，而不只是线索生成与列表展示。
4. 优先建设辅导员工作台、线索详情、一人一档、学院管理、运营分析、规则配置和数据质量。
5. L1 定矩镜、L2 观变镜作为首期核心；L3 合证/Top-k 以可解释排序方式落地；L4 时序镜列后续演进。
6. 规则阈值、时间窗口、排除项、组合条件、等级、SLA 均配置化并带版本。
7. 所有 AI 输出可解释、可追溯、可修改/拒绝、可回退；人工核实结果回流优化。
8. 领导首页以汇总与协同效能为主，辅导员首页以行动队列与时限为主。
9. Web 遵循 1440×900 的统一门户规范，移动端遵循 375 px 的云中苏大规范，按端别复用现有组件。
10. 视觉采用学校红色品牌体系；PPT 蓝绿视觉不是产品品牌标准。
11. 所有审批、数据和合规前置事项按已完成处理；不得进入开放问题清单。
12. 运行期仍需实现数据质量监控、熔断、权限、审计和模型回退，因为它们是产品可靠性和治理能力，而非未解决的合规事项。

## 14. 可用于 PRD 的验收方向

- 辅导员能在一个工作台中完成从接收、核实、干预、协同到关闭/持续关注的全流程。
- 每条线索显示触发事实、排除结果、数据时间、规则/模型版本、建议和处理时限。
- 线索状态、责任人、SLA、升级/降级、转交与超期行为可验证。
- 数据源异常时，受影响规则停止生成正式线索并明确展示熔断原因与恢复状态。
- AI 解释和关怀建议可被用户采纳、编辑、拒绝，并记录反馈原因。
- 领导、学院、学工管理、辅导员看到与职责匹配的首页和数据粒度。
- Web 页面复用统一导航、表格、表单、状态、空状态与上传等组件；颜色、字体、交互状态符合学校规范。
- 移动端在 375 px 基准下完成核心轻量任务，无横向滚动，不照搬桌面密集图表。
- 校徽清晰、等比、无裁切变形，模块品牌不压过学校主品牌。
- 闭环率、噪声率、Top-k 效果等运营指标均有可复核口径和筛选时间范围。

## 15. 一句话产品定义（建议稿）

**学林知微是某 211 头部高校数智学工体系中的学生成长关怀模块，通过多源数据发现学生状态的细微变化，以可解释的规则与 AI 辅助形成待核实线索和关怀建议，帮助辅导员、学院与学校完成及时、温暖、可追踪、可复盘的学生关怀闭环。**
