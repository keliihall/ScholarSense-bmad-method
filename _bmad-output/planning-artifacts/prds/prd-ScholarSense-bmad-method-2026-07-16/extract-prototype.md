# 前端原型提取摘要

## 1. 原型性质

- `docs/input/原型/frontend` 是 Vue 3 + TypeScript + Element Plus + ECharts 的纯前端交互原型，包含 mock 数据与前端登录守卫，不应视为生产系统实现。
- Web 管理端部署基路径为 `/scholarsense/`，定位为学校数智学工大系统中的“学林知微”模块。
- 原型表达的核心产品原则是：输出“需人工核实的关怀线索”，不把信号直接解释为学生结论；系统负责排序、解释、协同和闭环，责任人员完成判断。

## 2. 用户与数据范围

原型定义七类角色：辅导员、学院管理、学工管理、校级领导、协同部门、系统运维、数据责任人。

- 辅导员：本人责任学生，核心入口为工作台、线索和一人一档。
- 学院管理：本学院汇总、线索督办与协同。
- 学工管理：全校治理、规则、质量、运营与汇总。
- 校级领导：领导驾驶舱，默认汇总态势。
- 协同部门：心理、资助、教务、保卫、后勤等转介接收与结果回填。
- 系统运维：权限、审计、模型运行，不默认查看业务明细。
- 数据责任人：数据源质量与熔断处置。

## 3. 信息架构与页面能力

### 关怀工作台

- 辅导员工作台：待核实、处理中、今日已跟进、本院超期、今日核实容量、Top-k 优先级线索。
- 线索列表与详情：学生概况、风险评分构成、证据链、解释和关怀建议、核实反馈、噪声原因、关怀方式、转介与持续关注。
- 一人一档：责任学生筛选、基本信息、关爱标签、学业/经济/安全信号、个人基线、历史线索与跟进记录。
- 协同转介：按待接收、处理中、已回填、已关闭流转，支持目标部门、处理时间线、结果回填和导出。

### 治理与运营

- 成长总览：新增线索、重点关怀学生、闭环率、噪声率、趋势、类别与等级分布、学院对比。
- 规则配置：18 条学业/经济/安全规则宽表，含生产、灰度、熔断状态与规则详情。
- 数据质量：数据源责任部门、更新频率、主键完整性、连续性、覆盖率、最近更新、熔断。
- 周报月报：新增、闭环、噪声、超期、学院对比、噪声原因与协同成效。
- 学生标签：身份标签和关爱标签的来源、用途、敏感性与授权范围。
- 自然语言问数：面向汇总态势的统计查询。

### 领导与平台治理

- 领导驾驶舱：今日新增、重点关怀、闭环率、噪声率、待核实、超期及学院横向对比。
- 权限与脱敏：角色权限矩阵、数据范围和字段脱敏配置。
- 审计留痕：登录、查看、导出、跟进、规则修改、白名单、下钻和越权尝试。
- 模型与质量治理：数据质量、效果评估、分群表现、人工复核、灰度发布和稳定基线回退。
- 合规检查页：原型中的旧建设关卡。根据用户最新指令，审批与合规前置事项已完成；正式 PRD 不再把该页作为上线阻塞流程，但保留运行期权限、审计、数据质量和人工核实能力作为产品能力。

## 4. 核心领域对象

- 学生、责任关系、身份标签、关爱标签。
- 关怀线索、证据项、个人基线、关怀等级、线索状态、核实反馈。
- 规则与版本、数据源质量状态、协同转介工单、审计日志、报表、模型版本。
- 线索三类：学业、经济、安全；状态包括待核实、处理中、已跟进、超期、已关闭、已熔断。

## 5. 规则产品化线索

- 原型包含 18 条规则：学业 9 条、经济 3 条、安全 6 条。
- 学业/安全规则原型采用身份系数，经济规则不采用身份系数；此机制需要在正式 PRD 中作为业务规则能力而非写死算法实现，并确保可配置、可解释、可灰度、可熔断、可回退。
- 原型同时存在固定阈值规则、个人历史基线、多源合证与 Top-k 排序，正式 PRD需区分 MVP 必备能力与后续增强能力。

## 6. 可复用的体验承诺

- “从细微变化中，把关爱更早送到。”
- 入口统一、角色可见、信息按工作任务组织。
- 每条线索可解释、可追溯，并显式展示数据可信度、排除条件和时限。
- 以人工核实、关怀跟进、跨部门协同和结果回填构成闭环。
- 管理端以运行效能和资源调配为中心，避免把汇总指标误当学生个体结论。

## 7. 需要在正式 PRD 中纠偏的原型限制

- 原型把大量建设期合规文案写入页面。正式产品应保留必要的运行期提示，但避免让主任务被重复说明淹没。
- 原型角色数量与早期需求清单“五类角色”不一致，应以业务职责整合为七类角色模型，并在权限系统中允许组合授权。
- 原型中的“AI 解释”“自然语言问数”“模型治理”不代表均进入 MVP；需依据材料中的分期策略明确范围。
- 原型仅验证信息架构和交互意图，不构成后端、数据、集成、性能或上线就绪证明。
