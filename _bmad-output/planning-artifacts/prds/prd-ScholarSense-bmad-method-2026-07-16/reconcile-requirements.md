# PRD 输入对账报告：需求清单与两份 DOCX

## 1. 对账范围与结论

本报告对照以下输入与当前产物：

- `docs/input/文档/需求列表.md`（需求清单 MD）
- `extract-requirements.md` 中提取的 `学林知微需求列表--20260706.docx`（需求清单 DOCX）
- `extract-requirements.md` 中提取的 `【第三版】（20260612）“数智学工”系统风险预警与干预功能设计方案(2).docx`（设计方案 DOCX）
- 当前 `prd.md` 与 `addendum.md`

总体判断：当前 PRD 已覆盖产品定位、角色、数据底座、规则/基线、四类主场景、工作台、一人一档、转介、学院治理、驾驶舱、运营复盘、模型治理、审计和主要 NFR，且正确落实了“审批、数据提供与合规事项均已解决”的用户前提，没有把这些历史事项重新列为阻塞项。仍有 **2 个高优先级流程/语义缺口、3 个中高优先级业务基线缺口，以及若干需补充来源追溯或假设标识的量化裁决**。

## 2. 最重要差距

### G-1（高）候选线索到正式线索的人工初审环节缺失，且术语与流程互相矛盾

**来源要求：** 设计方案及 `extract-requirements.md` 明确“系统输出候选线索；正式进入预警/关怀库前必须由辅导员人工初审筛选”，端到端流程也把“辅导员初审与核实”置于候选生成之后。

**当前 PRD：**

- §5 将候选线索定义为“尚未通过人工初审”，却将正式线索定义为“通过质量门槛并进入责任人员工作队列”，没有把“通过人工初审”写入正式线索定义。
- FR-25 只执行质量、排除和白名单检查；通过检查后即进入辅导员正式队列，没有初审动作、责任人、时限、通过/驳回结果与驳回原因。
- FR-29、FR-30、FR-31 直接从“正式线索/待核实”开始，导致初审与核实可能被混为同一步，也无法验收“未经辅导员初审自动入库事件为零”的来源护栏。

**建议修正：** 明确二选一并贯穿术语、状态机、指标与用户旅程：

1. 保留两阶段：候选池 → 辅导员初审（接纳/驳回/合并）→ 正式线索 → 核实/跟进；补充初审队列、时限、记录和护栏指标。
2. 若校方实际把“进入待核实队列”视为初审开始，则取消“正式进入前必须人工初审”的表述，把进入队列的对象仍称“候选线索”，直到首次人工接纳后转为正式线索。

该问题会直接影响 UX 信息架构、状态机、任务量口径、闭环率分母和 Epic 拆解，应在 Finalize 阶段解决。

### G-2（高）“身份等级影响什么”存在内部不一致，Fast path 裁决未完全落到 PRD

**来源要求与冲突：** 设计方案给出普通 ×1、一般关注 ×1.5、重点关注 ×2，并写作“风险行为得分 × 身份系数 = 预警得分”；经济类不加系数。需求清单强调关怀等级和跨类合证。`addendum.md` 已裁决为“身份等级只按校方规则影响指定学业/安全线索的核实优先级；不改变原始规则分数”。

**当前 PRD：**

- §5 术语表写“身份等级用于指定场景的核实优先级调整”。
- FR-18 却要求展示“原始分数、身份等级影响、最终等级”，语义上允许身份等级改变最终关怀等级。
- FR-19 又规定合证优先级不改变原始规则分数，但未说明身份系数落在哪个排序因子，也未保留来源中的 1/1.5/2 业务口径或说明这些系数已被正式废止。

**建议修正：** 统一为一个可计算、可验收的三层模型：规则原始分数 → 规则关怀等级；身份等级仅形成独立的优先级调整量（如校方仍采用 1/1.5/2，应注明作用对象是优先级而非事实结论）；跨类别合证再形成 Top-k 排序。若 Fast path 决定废止原系数，需在 addendum 中明确“替代/废止”及理由，避免实施团队按 DOCX 恢复旧算法。

### G-3（中高）具体规则目录被过度抽象，无法证明设计方案中的业务规则已被完整保留或有意废止

**来源要求：** 设计方案 DOCX 给出大量具体规则及边界，包括学业挂科/缓考/绩点下降/核心课与英语/缺勤/体测/社会实践/论文节点，经济无消费/有效餐次/餐均与环比下降，安全晚归晚出/24—48 小时无活动/宿舍进出/疑似校外住宿/夜间网络时段，以及辅导员进宿舍的有效统计口径。

**当前 PRD：** FR-14、FR-20 至 FR-24 只规定“规则可配置”和场景级结果；除 24/48 小时、28/56 日等少量参数外，未列出上述已给定规则，也没有指向一份“获批规则库基线”。`addendum.md` 只处理经济增分量纲和心理咨询两处冲突，未保留其余规则。

**影响：** 研发可实现一个通用规则引擎，却仍无法依据 PRD 验收“需要配置哪些首期规则、触发/不触发样本是什么、原规则是否被替换”。这不是要求把所有公式塞进 PRD，而是当前正式需求缺少可追溯的业务配置基线。

**建议修正：** PRD 增加“首期规则包须覆盖的规则族及验收要求”，详细阈值/公式放入 addendum 或独立规则目录并由 PRD规范性引用；每条规则至少记录 ID、适用人群、数据源、窗口、阈值/分值、排除条件、频率、等级映射、正反测试样本和版本。对已裁决不采用的具体规则逐条标记“废止/后置”，不能静默丢失。

### G-4（中高）推送/计算节奏的来源冲突被直接裁决，但“30 分钟”缺少假设标识与产品流程定义

**来源要求与冲突：** 设计方案倾向安全类每日计算/汇总推送、经济类每月 1 日计算推送、学业按期末/期中/缓考节点；需求清单 MD 又写安全类近实时、个人基线/Top-k 每日至少一次。

**当前 PRD：** §9.1 直接规定“安全类数据到达平台后 30 分钟内完成候选计算；经济类至少每日增量更新、每月正式结算；学业按关键成绩节点计算”。其中经济/学业已合理整合，但 30 分钟是新增量化裁决，未标 `[假设]`，也未定义：

- 30 分钟从源系统事件时间、平台接收时间还是完整批次到达时间起算；
- 是完成计算、进入候选池、进入正式队列还是完成公共消息通知；
- 数据迟到、批次不完整、熔断恢复时如何计时；
- 安全事项是否仍有每日汇总，如何避免实时与汇总重复提醒。

**建议修正：** 将 30 分钟列入 Fast path 假设索引或引用校方既定 SLA，并分别定义“数据新鲜度、候选计算完成、人工队列可见、通知送达”四个时点。明确实时/准实时与每日汇总的去重关系。

### G-5（中）学业推送节点与“入学适应期”规则语义未完全落地

**来源要求：** 设计方案存在学业“每学期 2 次”与“每学期 3 次（含期中）”的冲突；还给出 9—10 月面向大一/研一的入学适应期规则语义（心理红色信号叠加凌晨网络连接）。

**当前 PRD：**

- 仅写“学业按关键成绩节点计算”，没有裁决关键节点具体是期中、期末、缓考中的哪几个，也无最低频率/完成时限。
- FR-24 把入学适应期抽象为可配置专项，只保留人群、起止时间和排序策略；没有保留“心理授权信号 + 作息变化的合证”这一来源定性语义。

**建议修正：** 把节点清单和节点后完成计算的时限写为规则包基线；对入学适应期至少保留“新生、9—10 月、授权关怀信号与作息变化合证、只作人工核实线索”的能力语义，具体阈值仍可配置。

## 3. 分输入对账

### 3.1 `需求列表.md`

**已覆盖：** F-001 至 F-009 的统一身份、RBAC、责任范围、运维隔离、字段脱敏和首期无学生端；D-001 至 D-017 的 P0/P1 类别、主键映射、质量面板、熔断及 56 日历史；R/C/S/W/P/T/RC/I/L/OP/ST/AI/MG/AU 各能力簇；性能容量、安全、审计、可维护与主要集成；分期里程碑。

**遗漏/弱化：**

- W-001 的“学院超期统计”在辅导员首页被收敛为本人超期；若来源确指辅导员可见学院统计，需明确保留还是因权限最小化而废止。
- C-004 把“已熔断”列为线索状态；PRD 同样保留，但 FR-12 又说熔断时不生成正式线索。应明确“已熔断”只适用于已存在且证据后来失效的线索，还是规则状态，避免状态机歧义。
- D-013 要求每个数据源维护“授权边界”；FR-9 只列责任部门、口径、主键、频率、覆盖和依赖规则，未把授权边界列为数据源目录的验收字段。
- OP-003“规则校准建议”在 FR-47 中主要变成效果分析和创建校准任务，缺少建议如何产生、由谁确认以及不得自动发布规则的要求。
- IN-005/需求清单 DOCX 的具体通知入口被统一裁决为数智学工公共消息路由；该方向符合模块定位，但需明确公共消息能力至少覆盖任务创建、临期、超期、退回、数据熔断等事件，并给出送达/失败状态，而非只写依赖名称。

### 3.2 `学林知微需求列表--20260706.docx` 提取结果

**已覆盖：** DOCX 对 MD 的大多数功能扩展均已进入 PRD，尤其是移动轻量处置、协同回填、模型治理、驾驶舱、审计和数智学工集成边界。

**遗漏/需确认裁决：**

- DOCX 指定“云中苏大 App”的通知/入口语义被抽象为“数智学工公共消息与学校统一移动入口”。若云中苏大就是已确定的统一移动入口，PRD 应写明别名/映射，避免交付方误认为另一套入口。
- DOCX 与 MD 的转介对象差异已在 addendum 裁决为心理、资助、教务、保卫、学院五类，家长为线下动作、导师为协同参与者；该裁决已基本正确落地。
- DOCX 中具体首页字段、页面功能和信息架构若已由原型承载，PRD宜明确哪些属于规范性需求、哪些仅为原型参考；当前只笼统把原型列为输入，容易让 UX 团队误判原型约束力。

### 3.3 `设计方案 DOCX` 提取结果

**已覆盖：** 学业/经济/安全三类场景、身份等级与业务标签分离、规则与个人基线、多线索合证、人工核实、差异化节奏、辅导员进宿舍仅统计不预警、心理咨询次数不自动计分、关怀语言和禁止自动定性。

**主要缺口：**

- 具体规则目录及边界条件未形成规范性引用（见 G-3）。
- 候选线索必须经人工初审再进入正式线索库的流程未实现（见 G-1）。
- 身份系数是改变分数/等级还是只改变优先级未统一（见 G-2）。
- 学业推送 2 次/3 次矛盾未裁决（见 G-5）。
- “辅导员进宿舍”来源有明确有效记录口径（一进一出且间隔大于 5 分钟）、周度/月度统计维度；PRD 只在 P1 数据与非目标中提及，不包含统计功能和验收结果，属于明确功能遗漏。建议新增工作纪实统计能力，仍明确不得触发学生线索。

## 4. 量化与流程缺口清单

以下不是合规阻塞项，而是需要在正式产品基线中明确或标注为 Fast path 假设的交付参数：

| 项目 | 当前状态 | 需补充的可验收定义 |
|---|---|---|
| 候选初审 | 缺失 | 初审角色、队列、时限、接纳/驳回/合并、理由、审计、转正式条件 |
| I/II/III 时限 | 已给 5 工作日/48h/24h | 起算点、暂停/恢复、节假日、责任转移、超期口径 |

## 5. 定性语义保留检查

**保留良好：** “线索而非结论”“关怀而非风险定性”“证据先于建议”“个人变化优先”“不贴贫困标签”“网络数据只回答作息变化”“辅导员为最终核实人”“管理视图不制造机械排名”“模块不重复建设数智学工公共能力”。这些来源中的关键语气与价值约束均已在 PRD 主叙事、非目标和反向指标中得到强化。

**仍可能静默丢失：**

- “全体学生全域排查 + 重点群体精准摸排并行”在愿景中未形成明确功能/覆盖率要求；当前更偏场景化候选生成。建议在范围或规则运营中明确两种运行模式及其覆盖统计。
- “候选名单先人工初审，再成为正式关怀事项”的审慎语义没有进入流程（G-1）。
- 入学适应期的“授权关怀信号与作息变化合证”被抽象成普通季节专项（G-5）。
- 设计方案中的具体学业/经济/安全规则虽然不宜全部进入主体叙事，但若没有规范性规则目录，会失去“校方已有业务口径”这一重要语义。

## 6. 建议的 Finalize 处理顺序

1. 先裁决 G-1 的候选/正式线索状态机，否则后续 UX、指标和故事都会建立在歧义上。
2. 统一 G-2 的规则分数、身份影响、关怀等级与合证优先级四者关系。
3. 建立规范性“首期规则包”伴随物，逐条保留、替代或后置设计方案规则，解决 G-3/G-5。
4. 将安全 30 分钟、I 级 5 个工作日等来源未确认数字纳入假设索引，或注明校方既定 SLA 来源；补全起止口径。
5. 补入辅导员进宿舍统计、公共消息事件、数据源授权边界和全域/重点双模式等遗漏，再进入 reviewer 与 polish。

## 7. 合规前提核验

本次对账未提出任何新的审批、数据可得性或合规阻塞项。PRD 当前把相关事项写为“按学校已确定方案执行”的既定前提是正确的。报告中涉及权限、脱敏、审计、数据最小化、人工复核、保留删除和网络数据边界的内容，均属于已经获准后仍需落实的产品质量与验收要求，不是待审批问题。
