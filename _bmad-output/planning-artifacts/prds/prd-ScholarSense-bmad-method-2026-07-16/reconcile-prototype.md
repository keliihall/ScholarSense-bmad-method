# 输入对账：前端交互原型

## 1. 对账范围与判定原则

**输入：**

- `docs/input/原型/frontend/`
- `extract-prototype.md`

**目标：**

- `prd.md`
- `addendum.md`

本次按页面、角色、对象、状态、规则、反馈、治理能力七个维度核验。原型是 Vue 3 + TypeScript + Element Plus + ECharts 构成的纯前端演示，数据来自 `src/mock/*`；登录、权限、保存、导出、审批、状态变更、熔断、发布、回退、问数等大量动作仅在浏览器内显示消息或改变局部内存状态。因此：

1. 原型可证明信息架构、术语、页面意图和体验原则；
2. 原型不能证明后端能力、数据接入、集成、持久化、权限安全、审计不可篡改、性能、生产可用性或流程已落地；
3. 原型中的数值、阈值、角色可见性、按钮和模拟流程，只有得到其他正式输入支持时才能提升为 PRD 承诺；
4. PRD 对原型进行产品化收敛或纠偏时，以正式 PRD 为准，不应为了“还原原型”退回较弱语义。

## 2. 覆盖结论

总体结论：**高覆盖，可进入评审；未发现整页或核心领域对象被遗漏。** `prd.md` 已完整覆盖原型表达的主干，并明确将其定位为数智学工大系统中的模块，而非独立系统。尤其是统一入口、七类角色、线索—证据—核实—转介—回填闭环、数据质量熔断、规则生命周期、学院治理、驾驶舱、审计、模型/策略治理均已形成可验收要求。

### 2.2 角色与数据范围

七类角色在 PRD 中均有对应：辅导员、学院管理、学工管理、校级领导、协同部门、系统运维、数据责任人。PRD 比原型更准确地采用“角色 + 组织 + 责任学生 + 有效协同任务”组合授权，并明确运维默认不能查看业务明细。

需防止照抄两处原型宽授权：

- 原型路由允许协同角色进入“一人一档”，但生产语义必须受有效协同工单和最小任务范围约束，不能由菜单可见直接推导对象明细可见；PRD FR-5、FR-7 已正确收紧。
- 原型把领导的数据范围标记为 `all`，同时在页面声明默认仅汇总。生产实现必须以 FR-45 的“默认不展示学生明细”为准，不能把 `dataScope: all` 解释为全校个体明细权限。

### 2.3 领域对象与状态

学生、责任关系、身份等级/业务标签、正式线索、证据项、个人基线、规则版本、质量状态、转介工单、审计日志、报表、模型/策略版本均已覆盖。

PRD 对状态机做了必要增强：

- 线索：原型为待核实、处理中、已跟进、超期、已关闭、已熔断；PRD 增加“待观察”，并将超期事实保留。
- 转介：原型为待接收、处理中、已回填、已关闭；PRD 增加“待补充、退回”，并明确原线索由辅导员最终关闭。
- 规则：原型类型仅含草稿、灰度、生产、熔断；PRD 增加测试、停用、回退等生产状态。

这些增强不属于无依据扩张，而是让原型已表达的人工核实、持续关注、协同回填和灰度回退能够形成可验收闭环。

### 2.4 规则、反馈与治理

- 原型 18 条规则及学业/经济/安全分类已被场景、规则配置和规则生命周期能力吸收。PRD 未把 `6/9/12`、`×1/×1.5/×2` 等演示常量固化为不可变算法，而要求版本化、可配置、可测试、可解释、可灰度、可熔断、可回退，处理正确。
- 经济规则不使用身份系数、心理咨询次数不自动计分、跨类别信号只提升核实优先级而不改写原始分数，均已在 FR-18、FR-19 和 `addendum.md` 中明确。
- 联系情况、属实/噪声/待观察、噪声原因、关怀方式、是否转介、持续关注和备注已由 FR-32、FR-33 覆盖。
- 权限脱敏、数据质量、审计、规则运营复盘、模型/策略发布关卡与稳定基线回退已覆盖；旧“合规检查页”没有被错误提升为新的产品主流程。

## 3. 需要处置的差距

### GAP-P1（高）：通知计算/推送节奏与抑制策略尚未形成可验收口径

原型 `constants.ts` 表达了学业、经济、安全三类不同的计算与推送频率，PRD FR-4 已覆盖消息/待办联动，NFR §9.1 也定义了部分数据计算时效，但未明确：

- 实时事项与日汇总/月汇总如何区分；
- 同一学生多线索合证后是合并通知还是逐条通知；
- 已读、认领、处理中、待观察状态如何抑制重复提醒；
- 临期、超期升级通知的节奏与接收层级。

这会直接影响一线负担、Top-k 队列和 SM-C2。建议在 PRD 或后续 UX/架构验收口径中补充“事件生成、聚合、去重、升级、关闭同步”的通知策略；不要直接照抄原型的固定频率常量。

### GAP-P2（高）：领导明细下钻的运行期授权流程边界不够明确

原型驾驶舱提供“申请授权下钻”的交互意图；PRD FR-45 只规定默认不展示学生明细，FR-5/FR-7 规定按任务授权，§8 又明确不重复建设通用审批。三者方向一致，但尚未明确以下产品合同：

- 领导是否允许申请个体下钻，还是始终仅看汇总；
- 若允许，复用数智学工通用审批还是由有效业务任务直接授予；
- 临时权限的对象范围、期限、撤销与审计事件。

这不是重新打开项目合规审批，而是运行期访问控制。建议产品负责人在 UX/架构前明确一种模式；不得把原型弹出的提示框视为已经确定或已经实现的审批流程。

### GAP-P3（高）：原型的核实提交动作错误压平状态，UX 实现必须服从 PRD

`ClueDetailView.vue` 无论选择“属实、噪声、待观察”、是否转介或是否持续关注，演示提交后都统一提示状态流转为“已跟进”；页面按钮还使用“提交反馈并闭环”。这与 PRD FR-29、FR-32、FR-33、FR-39～FR-41 的状态语义不一致。

正式 UX 至少应区分：

- 待观察：进入观察周期并生成复查任务，不能直接视为闭环；
- 发起转介：原线索保留处理中/等待协同结果，转介关闭也不自动关闭原线索；
- 噪声：记录噪声原因并进入可复盘状态；
- 属实且已完成关怀动作：方可进入已跟进或满足条件后关闭。

此处 PRD 已正确，后续不能以“还原原型”为由退回单一状态。

### GAP-P4（中）：导出与配置按钮多数只有 mock 提示，不能自动视为范围承诺

原型包含线索列表、一人一档、转介流转、总览、权限矩阵、审计、治理报告、标签口径等多类导出按钮，也包含新增标签、规则状态切换、熔断恢复、模型回退等演示操作。其中不少函数明确写有“演示”“Mock 反馈”“不写库”，或仅显示 `ElMessage`。

PRD 已明确承诺周报/月报授权导出、全局审计和异步大范围导出的非功能要求，但没有逐一承诺所有页面均可导出或每个演示按钮均进入首期。建议后续 UX 逐项建立导出矩阵：业务目的、角色、字段范围、格式、审批/授权、异步任务、有效期、水印及审计；没有业务依据的按钮应删除，而不是补做后端以迁就原型。

### GAP-P5（中）：标签“新增”交互不能被解释为任意用户即时创建标签

原型标签页的“新增标签”仅是 mock 对话框，不写入数据，也没有完整的审核、生效、失效、冲突处理和计算参与规则。PRD FR-37、FR-38 已正确要求来源、用途、角色、有效期、审核、争议和暂停参与计算。

后续 UX 应将原型按钮改造为受控的标签定义/变更申请，或在首期不开放创建，仅同步权威来源。不能把弹窗表单提升为“学工或协同角色可直接创建并生效任意标签”的承诺。

## 4. 不应提升为正式承诺的原型内容

以下内容仅为演示细节或已被 PRD 有意纠偏：

- 前端选择角色登录、前端路由守卫和 `/scholarsense/` 本地部署配置，不代表生产身份认证与权限控制实现。
- mock 数据中的人数、KPI、学院排名、时间、姓名、学生记录、Precision@K/Recall@K、噪声率和公平性结论。
- `6/9/12` 分级阈值、固定身份系数和页面内硬编码推送频次；应以已发布规则版本为准。
- “AI 解释”标题不表示 MVP 依赖生成式模型；PRD 的结构化模板解释基线及智能后置分期应保持。
- 自然语言问数和模型治理页面存在，不表示它们进入 MVP；PRD §7.3 的后续增强定位应保持。
- 合规检查关卡的未完成状态、提醒推进和“关卡未过不得生产”等建设期 mock 状态；用户已明确前置事项全部解决。
- “家长”作为转介目标；`addendum.md` 已正确界定联系家长是辅导员线下关怀动作，家长不是首期系统用户或系统间转介部门。
- 页面上的导出成功、留痕成功、熔断成功、回退成功、配置成功等消息，不证明真实数据持久化、权限验证、审计写入或后台任务已经存在。

## 5. 建议处置顺序

1. 在正式 PRD 定稿前决定 GAP-P1 的通知产品合同；它影响 FR-4、FR-19、FR-29、FR-31 以及反向负担指标。
2. 在 UX/架构启动前决定 GAP-P2 的领导下钻模式，明确这是运行期授权而非重新讨论前置合规。
3. 将 GAP-P3 作为 UX 状态机验收红线，确保不复制原型的“一键已跟进/闭环”。
4. 将 GAP-P4、GAP-P5 放入 UX 功能盘点和权限矩阵，不因原型存在按钮而自动扩展首期范围。

## 6. 最终判定

`prd.md` 与 `addendum.md` 对前端原型的核心意图覆盖充分，且在角色权限、状态机、生产治理和分期上普遍优于原型。当前没有必须新增页面或核心对象的阻塞性缺口。定稿前最值得补强的是通知与提醒合同；领导运行期下钻授权需形成明确选择。其余差异主要应通过 UX 纠偏和“mock 非承诺”边界控制解决。
