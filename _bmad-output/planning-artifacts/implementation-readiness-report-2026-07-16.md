## PRD 分析

### 功能需求

- FR1：从数智学工统一门户进入并复用学校 SSO。
- FR2：同步权威账号、角色、组织树和辅导员—学生责任关系。
- FR3：按七类职责提供角色化首页、菜单和待办。
- FR4：将候选、临期、超期、转介和质量异常幂等联动公共消息/待办。
- FR5：同时按角色、组织、责任学生和协同任务限制访问。
- FR6：按字段、角色、任务统一执行明文、脱敏、不可见投影。
- FR7：敏感信息仅在责任关系或有效协同任务期间授权。
- FR8：登录、读取、导出、关怀、治理、熔断和越权均全链路审计。
- FR9：展示 P0/P1 数据源责任、口径、主键、频率、覆盖和依赖规则。
- FR10：将多源标识唯一关联到学生，歧义隔离并可修复重算。
- FR11：计算完整性、连续性、覆盖率和新鲜度并固化质量快照。
- FR12：质量跌破门槛自动熔断，验证与样本重算后授权恢复且不补发失效线索。
- FR13：保留个人基线/复盘历史并支持受控、去重重算。
- FR14：配置阈值、时间窗、人群、条件、排除、等级、时限和解释模板并支持匿名样本测试。
- FR15：规则具备草稿、测试、灰度、生产、熔断、停用和回退生命周期。
- FR16：按场景形成并解释个人 28/56 日或其他历史基线。
- FR17：计算前应用场景排除和有期限、有原因、可审计的白名单。
- FR18：分别保存/展示规则分数、关怀等级、身份排序调整和工作优先级。
- FR19：候选与正式线索队列按等级、时效、可信度形成可解释 Top-k。
- FR20：住宿安全场景结合门禁、请假、住宿、课表、设备状态生成候选并受质量熔断。
- FR21：经济压力场景基于本人消费历史变化和有效餐次/请假/离校排除，禁止污名化标签。
- FR22：毕业季组合独立信号形成专项优先队列，不复制原线索并支持 7 日观察。
- FR23：夜间作息仅使用网络会话汇总，连续异常才生成候选且禁止内容数据。
- FR24：配置有期限的入学、考试、毕业等专项队列和学业日历触发。
- FR25：经质量、排除、白名单、去重后生成候选，支持接纳/拒绝/合并并从生成时启动时钟。
- FR26：正式线索包含完整字段且列表/详情口径一致、变化留历史。
- FR27：正式线索提供来源、指标、当前值、基线、变化、窗口、排除和质量的结构化证据链。
- FR28：按模板生成区分事实、判断、非强制建议的解释并收集可用性反馈。
- FR29：正式线索按定义状态流转，超期、质量、规则运行状态正交且时钟不被接纳/转移重置。
- FR30：工作台分别展示候选初审、待核实、处理中、已跟进、超期和容量。
- FR31：Top-k 支持按优先级、截止、类别、状态和学生筛选并解释排序。
- FR32：记录联系、核实、噪声、关怀、转介、观察和备注，按结果动态必填并自动带入已有信息。
- FR33：设置观察周期、关注点和复查时间，聚合新证据并到期产生单一复查任务。
- FR34：责任变化/代办时转移未完成线索、保留交接和原时钟，无接收人进入异常队列。
- FR35：按最小必要原则展示学生基础档案、来源和更新时间。
- FR36：时间线区分机器信号、人工核实、协同结果，噪声不作属实历史。
- FR37：身份等级与业务标签分别治理来源、用途、可见性、有效期和审核。
- FR38：支持标签错误/过期/不适用反馈，争议可暂停计算且结果可追溯。
- FR39：从正式线索向校内部门发起含最小必要证据的唯一转介。
- FR40：转介支持待接收、处理中、待补充、已回填、已关闭和退回状态。
- FR41：协同人员回填结果到原线索，转介关闭不自动关闭原线索。
- FR42：与既有部门系统幂等交换关联工单状态和结果，失败可持久重试。
- FR43：学院总览展示运行、时效、闭环、噪声和场景指标并授权下钻。
- FR44：对临期、超期、无责任事项发起并跟踪督办。
- FR45：领导驾驶舱只展示带周期/更新时间的全校汇总、趋势、学院对比和资源压力，不临时下钻个体。
- FR46：按统一口径生成可审计导出的周报/月报。
- FR47：按规则、场景、学院和人群复盘命中、有效、噪声、排除、熔断和反馈并创建校准任务。
- FR48：首期结构化解释不依赖生成式服务，模板版本随线索保存。
- FR49：授权灰度生成可采纳/编辑/拒绝的非强制建议，不自动改变等级或触发行动。
- FR50：自然语言仅查询授权、已定义汇总指标并展示口径、时间、更新时间。
- FR51：增强策略发布须通过质量、效果、分群和人工复核关卡及样本要求。
- FR52：保留稳定规则基线并可审计回退，不影响工作台、历史或未完成任务。
- FR53：监控数据任务、规则、消息、接口和关键用户路径并按角色隔离业务明细。
- FR54：依赖故障时保留任务、幂等重试、显示降级/数据时间且恢复不重复。
- FR55：移动端完成待办摘要、联系、轻量核实、观察和协同进度，并与 Web 共享唯一状态。
- FR56：按明确配对规则统计进宿舍工作，严格隔离于学生研判和辅导员绩效结论。
- FR57：按学业节点生成带学期/批次/版本/时间的去重候选并与毕业季合证。
- FR58：同一学生 7 日内两类独立线索提升合证优先级，不相加分数、不改等级，失效时重算留因。

### 非功能需求

- NFR1：容量支持约 5 万学生、每日数百万事件及 500—1000 峰值并发。
- NFR2：工作台、线索列表和详情正常负载 P95≤2 秒。
- NFR3：看板常用筛选 P95≤3 秒；大导出异步执行。
- NFR4：安全事件入库后 30 分钟内生成候选，候选后 5 分钟内生成待办；学业/经济按规定节奏。
- NFR5：业务时段月可用性≥99.9%，RPO≤15 分钟，RTO≤2 小时。
- NFR6：单源、生成式或报表故障不得阻断 SSO、人工工作台和既有任务。
- NFR7：规则、策略、集成操作支持幂等、重试、灰度和回退。
- NFR8：P0 标识完整率≥99.5%、核心字段覆盖≥98%、新鲜度≥99%，不达标必须熔断。
- NFR9：正式线索保存质量快照、规则版本和数据截止时间。
- NFR10：传输保护、敏感字段加密、集中密钥、输入服务端校验、参数化查询和上下文输出处理。
- NFR11：审计防篡改且至少保留 6 个月；成功越权、非授权明文导出、绕审计操作为零。
- NFR12：Web 适配统一门户/UI，验收 1440×900 与 1366×768、Edge/Chrome 最新两个稳定大版本。
- NFR13：移动适配学校 App iOS/Android WebView，375px 无横向滚动。
- NFR14：Web/移动状态同步 P95≤5 秒且不能重复提交。
- NFR15：WCAG 2.2 AA；状态不只靠颜色，表格键盘可用，图表提供等价表格/数值。
- NFR16：文案关怀、中性、可解释、非诊断，禁止自动重大不利决定。
- NFR17：关键链路贯通 traceId，遥测不记录学生明文、证据正文或密钥。
- NFR18：规则/配置受控变更不依赖代码发布；指标定义、负责人、周期、版本可追溯。
- NFR19：身份/责任增量同步 15 分钟内生效，每日全量对账。
- NFR20：周报/月报在截止后 2/4 小时内可用。
- NFR21：单条核实人工录入中位≤3 分钟并自动带入可用信息。
- NFR22：重复正式线索<1%，缺最低证据进入正式队列数量为 0。
- NFR23：候选量 P95≤初审容量 120%，超时积压不得连续两日增长。
- NFR24：性能压测使用 1000 并发/每日数百万事件模型并记录分布与计算窗口。
- NFR25：枚举、规则、浏览器矩阵和公共能力版本进入带生效日期的发布基线。

**非功能需求总数：25。**

## UX 对齐评估

### UX 文档状态

### 对齐良好的部分

- UX UJ-1—UJ-6 与 PRD 六条旅程一致，候选初审、正式线索、转介、学院督办、质量恢复、领导汇总的闭合点没有改变。
- 事实—规则判断—建议分层、线索而非结论、质量熔断、人工重大决定、最小必要字段与非污名化语言同时得到 PRD 和 AD-8/9/18/21 支持。
- P95 2 秒/3 秒、跨端 5 秒、375px、1440×900、1366×768、WCAG 2.2 AA 等指标在 PRD、UX、AD-17/22 和 Stories 中基本一致。
- 候选/线索双视图、截止时钟不重置、超期/质量/规则运行正交、转介关闭不自动关闭原线索与 AD-19/20 一致。
- `chart-with-table`、服务端字段投影、恢复观察、异步导出与可重建读模型得到 AD-9/11/13/17/24 支撑。

### 对齐问题

1. **PRD“无开放问题”与 UX/架构开放项冲突（高）**：PRD §15 声称不存在阻塞 UX、架构或 Epic 的开放问题，但 UX 仍询问高风险动作门禁、角色叠加/工作视角、768/1024 断点、关怀等级措辞，DESIGN 仍缺校徽矢量/安全区、控件尺寸、栅格、移动辅助色语义；Architecture 也将这些列为 Deferred。它们不会阻止领域开发，但会阻止相关 UI、安全审批和正式验收冻结。
2. **移动触控目标仍是假设（中）**：UX 规定移动 44×44 CSS px 为 `[ASSUMPTION]`，SPEC 也把它列为开放问题；Story 7.1 却将其写成无条件验收值。应明确“临时产品基线，校方确认后版本化变更”，否则验收权威不清。
3. **响应断点仍是假设（中）**：UX 的 768/1024 行为阈值非校规，Architecture Deferred；Story 1.1/7.1 覆盖端点视口，但没有要求在校方容器中验证并冻结中间断点版本。
4. **高风险动作映射未解决（高）**：UX 要求区分即时确认、二次认证、审批、双人复核；AD-23 只定义统一策略端口和默认拒绝，具体 actionType→门禁等级未定。涉及敏感导出、临时授权、规则发布/回退、白名单、熔断恢复和批量治理的 Story 只能测试“默认拒绝”，无法测试最终批准路径。
5. **多角色叠加行为未解决（中）**：UX 禁止模拟换角，但同一自然人多个角色时是否允许工作视角切换未定；Story 1.4 仅写“常设授权”，不足以验证菜单组合、冲突职责和默认落点。

### 警告

- PRD 将校方 UI/安全方案描述为“已确定”，UX/架构则明确存在未提供契约。实施前应把这些从隐性外部依赖转成带责任人、截止日期和临时默认值的发布阻塞清单。
- DESIGN 的 2/4/8px 圆角与部分移动行为为产品假设；应保持 token 化，不能散落为组件常量。

## 总结与建议

### 总体实施就绪状态

**NOT READY（暂不具备实施就绪条件）**

需求覆盖本身完整（58/58），PRD、UX 与架构的大方向高度一致；但 Story 顺序存在真实前向依赖，多个 Story 超出单代理实施规模，并且规则验收基线、高风险动作门禁、角色叠加与部分 UI 验收权威仍未冻结。若当前直接进入 Sprint Planning，开发团队无法仅依据现有 Story 独立实现和验收。

### 必须立即处理的严重问题

1. **消除 5 项前向依赖**：重构 Story 1.3、1.5、1.6、Epic 2 的规则/线索依赖，以及 Story 4.2→4.4 的反向顺序。
2. **补齐规则验收权威源**：提供 PRD 所引用的“附录 D”或等价版本化规则目录，至少包含各场景规则 ID、版本、阈值/窗口/排除、量纲、边界样本和生效日期；在此之前经济规则12与所有“按发布规则”条件不可验收。
3. **冻结高风险动作矩阵**：为敏感导出、临时授权、白名单、规则/策略发布回退、熔断恢复和批量治理明确 deny/二次认证/审批/双人复核映射、责任人和测试证据。
4. **拆分 4 个超大 Story**：Story 1.1、1.2、2.5、3.10 必须拆小；Story 4.5 建议拆分。每个新 Story 应有单一聚合/用户结果、独立测试边界和明确前置。
5. **补齐最小工程/CI 基线**：明确可重复构建、单元/集成测试、依赖与漏洞检查、制品产出、环境配置隔离及失败门禁。

### 推荐修复顺序

1. 运行 `bmad-correct-course`，以本报告为输入重排 Epic/Story，而不是直接手工零散修补。
2. 由产品、架构、安全、UX 共同形成三个冻结附件：规则目录、高风险动作矩阵、宿主/角色/UI 未决项决策表；每项指定 owner、deadline、临时默认值和发布阻塞级别。
3. 重写拆分后的 Story 验收标准：每个场景使用分行 Given/When/Then/And 和 AC ID，消除“可追踪/稳定/仍有时效”等未定义词，性能项绑定 AD-22 的负载与测量边界。
4. 为关键错误路径增加验收：契约责任缺失、样本不足、事件乱序/间隙、分群不足、WebView 不兼容、审批过期/版本变化。
