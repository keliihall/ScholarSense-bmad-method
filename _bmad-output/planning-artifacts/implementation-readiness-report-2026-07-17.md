## PRD 分析

### 功能需求

- FR1：所有用户可从数智学工统一门户进入并复用学校 SSO；重认证后返回原目标。
- FR2：同步权威账号、角色、组织树和辅导员—学生责任关系，保留来源版本和更新时间。
- FR3：按七类职责提供角色化首页、菜单和待办；未授权路由拒绝并审计。
- FR4：将新候选、临期、超期、转介和质量异常幂等联动公共消息/待办，支持更新、升级、关闭与失败重试。
- FR5：同时按角色、组织、责任学生和协同任务限制数据访问。
- FR6：按字段、角色和任务统一执行明文、脱敏、不可见投影。
- FR7：敏感信息仅在责任关系或有效协同任务期间授权，任务结束即撤销。
- FR8：登录、读取、导出、关怀、治理、熔断和越权均形成不可变、可检索审计。
- FR9：展示 P0/P1 数据源责任、口径、主键、频率、覆盖和依赖规则。
- FR10：将多源标识唯一关联到学生，歧义隔离并可修复重算。
- FR11：计算完整性、连续性、覆盖率和新鲜度，并固化每次线索计算的质量快照。
- FR12：质量跌破门槛自动熔断依赖规则；验证和样本重算后授权恢复且不补发失效线索。
- FR13：保留个人基线/复盘历史并支持受控、去重重算。
- FR14：配置阈值、时间窗、人群、条件、排除、等级、时限和解释模板，并支持匿名样本测试。
- FR15：规则具备草稿、测试、灰度、生产、熔断、停用和回退生命周期。
- FR16：按场景形成并解释个人 28/56 日或其他历史基线，明确样本不足行为。
- FR17：计算前应用场景排除和有期限、有原因、可审计的白名单。
- FR18：分别保存/展示规则分数、关怀等级、身份排序调整和工作优先级。
- FR19：候选与正式线索队列按等级、时效和可信度形成可解释 Top-k。
- FR20：住宿安全场景结合门禁、请假、住宿、课表和设备状态生成候选并受质量熔断。
- FR21：经济压力场景基于本人消费历史变化和有效餐次/请假/离校排除，禁止污名化标签。
- FR22：毕业季组合独立信号形成专项优先队列，不复制原线索并支持 7 日观察。
- FR23：夜间作息仅使用网络会话汇总，连续异常才生成候选且禁止内容数据。
- FR24：配置有期限的入学、考试、毕业等专项队列和学业日历触发。
- FR25：经质量、排除、白名单、去重后生成候选，支持接纳/拒绝/合并并从生成时启动时钟。
- FR26：正式线索包含完整字段且列表/详情口径一致，变化保留历史。
- FR27：正式线索提供来源、指标、当前值、基线、变化、窗口、排除和质量的结构化证据链。
- FR28：按模板生成区分事实、判断、非强制建议的解释并收集可用性反馈。
- FR29：正式线索按定义状态流转，超期、质量、规则运行状态正交且时钟不被接纳/转移重置。
- FR30：工作台分别展示候选初审、待核实、处理中、已跟进、超期和容量。
- FR31：Top-k 支持按优先级、截止、类别、状态和学生筛选并解释排序。
- FR32：记录联系、核实、噪声、关怀、转介、观察和备注，按结果动态必填并自动带入已有信息。
- FR33：设置观察周期、关注点和复查时间，聚合新证据并到期产生单一复查任务。
- FR34：责任变化/代办时转移未完成线索、保留交接和原时钟，无接收人进入异常队列。
- FR35：按最小必要原则展示学生基础档案、来源和更新时间。
- FR36：时间线区分机器信号、人工核实、协同结果，噪声不作属实历史。
- FR37：身份等级与业务标签分别治理来源、用途、可见性、有效期和审核。
- FR38：支持标签错误/过期/不适用反馈，争议可暂停计算且结果可追溯。
- FR39：从正式线索向校内部门发起含最小必要证据的唯一转介。
- FR40：转介支持待接收、处理中、待补充、已回填、已关闭和退回状态，并通知临期/超期/退回。
- FR41：协同人员回填结果到原线索，转介关闭不自动关闭原线索。
- FR42：与既有部门系统幂等交换关联工单状态和结果，失败可持久重试。
- FR43：学院总览展示运行、时效、闭环、噪声和场景指标并授权下钻。
- FR44：对临期、超期、无责任事项发起并跟踪督办。
- FR45：领导驾驶舱只展示带周期/更新时间的全校汇总、趋势、学院对比和资源压力，不临时下钻个体。
- FR46：按统一口径生成可审计导出的周报/月报。
- FR47：按规则、场景、学院和人群复盘命中、有效、噪声、排除、熔断和反馈并创建校准任务。
- FR48：首期结构化解释不依赖生成式服务，模板版本随线索保存。
- FR49：授权灰度生成可采纳/编辑/拒绝的非强制建议，不自动改变等级或触发行动。
- FR50：自然语言仅查询授权、已定义汇总指标并展示口径、时间、更新时间。
- FR51：增强策略发布须通过质量、效果、分群和人工复核关卡及样本要求。
- FR52：保留稳定规则基线并可审计回退，不影响工作台、历史或未完成任务。
- FR53：监控数据任务、规则、消息、接口和关键用户路径并按角色隔离业务明细。
- FR54：依赖故障时保留任务、幂等重试、显示降级/数据时间且恢复不重复。
- FR55：移动端完成待办摘要、联系、轻量核实、观察和协同进度，并与 Web 共享唯一状态。
- FR56：按明确配对规则统计进宿舍工作，严格隔离于学生研判和辅导员绩效结论。
- FR57：按学业节点生成带学期/批次/版本/时间的去重候选并与毕业季合证。
- FR58：同一学生 7 日内两类独立线索提升合证优先级，不相加分数、不改等级，失效时重算留因。

**功能需求总数：58。**

### 非功能需求

- NFR1：容量支持约 5 万学生、每日数百万事件及 500—1000 峰值并发。
- NFR2：工作台、线索列表和详情正常负载 P95≤2 秒。
- NFR3：看板常用筛选 P95≤3 秒；大导出异步执行。
- NFR4：安全事件入库后 30 分钟内生成候选，候选后 5 分钟内生成待办；学业/经济按规定节奏。
- NFR5：业务时段月可用性≥99.9%，RPO≤15 分钟，RTO≤2 小时。
- NFR6：单源、生成式或报表故障不得阻断 SSO、人工工作台和既有任务。
- NFR7：规则、策略、集成操作支持幂等、重试、灰度和回退。
- NFR8：P0 标识完整率≥99.5%、核心字段覆盖≥98%、新鲜度≥99%，不达标必须熔断。
- NFR9：正式线索保存质量快照、规则版本和数据截止时间。
- NFR10：传输保护、敏感字段加密、集中密钥、服务端输入校验、参数化查询和上下文输出处理。
- NFR11：审计防篡改且至少保留 6 个月；成功越权、非授权明文导出、绕审计操作为零。
- NFR12：Web 适配统一门户/UI，验收 1440×900 与 1366×768、Edge/Chrome 最新两个稳定大版本。
- NFR13：移动适配学校 App iOS/Android WebView，375px 无横向滚动。
- NFR14：Web/移动状态同步 P95≤5 秒且不能重复提交。
- NFR15：WCAG 2.2 AA；状态不只靠颜色，表格键盘可用，图表提供等价表格/数值。
- NFR16：文案关怀、中性、可解释、非诊断，禁止自动重大不利决定。
- NFR17：关键链路贯通 traceId，遥测不记录学生明文、证据正文或密钥。
- NFR18：规则/配置受控变更不依赖代码发布；指标定义、负责人、周期、版本可追溯。
- NFR19：身份/责任增量同步 15 分钟内生效，每日全量对账。
- NFR20：周报/月报在截止后 2/4 小时内可用。
- NFR21：单条核实人工录入中位≤3 分钟并自动带入可用信息。
- NFR22：重复正式线索<1%，缺最低证据进入正式队列数量为 0。
- NFR23：候选量 P95≤初审容量 120%，超时积压不得连续两日增长。
- NFR24：性能压测使用 1000 并发/每日数百万事件模型并记录分布与计算窗口。
- NFR25：枚举、规则、浏览器矩阵和公共能力版本进入带生效日期的发布基线。

**非功能需求总数：25。**

### 其他要求与约束

- MVP、首期完整上线、后续增强分别绑定指定 FR 集合；FR49—FR52 后置。
- 学生和家长不是首期用户；不得重复建设门户、身份、消息、审批和源业务系统。
- A-1—A-6、版本化规则目录、高风险动作矩阵和未决事项决策表共同组成发布基线。
- 外部依赖必须登记业务/技术 owner、SLO、失败策略、补数/重试和对账方式。
- 试点要求两个培养层次结构不同的学院至少运行 8 周。
- 反向护栏覆盖重复率、无证据率、自动定性、越权、排序伤害、候选负荷和积压。

### PRD 完整性初评

PRD 的产品边界、六条用户旅程、58 个 FR、25 个 NFR、分期、指标、非目标与治理附件引用齐全。高优先级内部矛盾仍存在：§0 与 §14 宣称审批、数据和合规事项均已完成/解决，而 §15 明确存在 Story/发布阻塞级未决事项。该冲突会让实施者错误绕过 `open-decisions.md`，后续一致性评估必须将其列为阻塞性问题。

## UX 对齐评估

### UX 文档状态

已找到并完整读取 `DESIGN.md` 与 `EXPERIENCE.md`。两者分别定义视觉令牌/组件外观与信息架构/行为/状态/无障碍/关键旅程，并明确冲突时双脊柱优先于原型。

### 对齐良好的部分

- UJ-1—UJ-6、候选/正式线索分区、责任时钟、质量熔断、协同回填和汇总治理与 PRD 主流程一致。
- 服务端组合授权、字段三态、读时撤权、图表/表格同 DTO、异步作业、跨端唯一状态均有 AD-7—AD-17 支撑。
- 1440×900、1366×768、375px、P95 2/3/5 秒、WCAG 2.2 AA 与 PRD NFR、AD-17/AD-22 一致。
- “事实—规则判断—建议”、非污名化语言、禁止自动重大决定与 AD-18 一致。
- 领导驾驶舱无个体下钻、工作纪实用途隔离、夜间网络数据最小化与 AD-21 一致。

### 对齐问题

1. **经济旅程缺少 blocked 状态（高）**：UJ-2 直接描述完整经济线索和资助转介，但正式规则目录中 `ECON-012` 仍 blocked。UX 未定义规则未批准时的不可用入口、空状态或防止演示数据被误认为生产能力的行为。
2. **UJ-4 与 PRD/Architecture 关闭语义冲突（高）**：UX 规定“发起辅导员完成最终闭环后，工单进入已关闭”；PRD FR41 与 AD-20 只规定工单关闭不得自动关闭原线索，并未规定原线索关闭触发工单关闭。该双向耦合会导致两个聚合的 owner 和关闭权限不清。
3. **UX 引入未追踪功能（高）**：`EXPERIENCE.md` 的“审计与合规”要求“合规材料的上传、复核、退回和完成状态可追踪”，但 PRD FR8 只定义审计检索，Epics 无合规材料工作流 Story，Architecture 也没有该对象的 owner、状态机、存储或保留策略。应删除该 UX 范围或先新增 PRD/Architecture/Story。
4. **高风险矩阵被当作已冻结（高）**：UX Open Questions 声称矩阵 v1.0 已冻结；但 `high-risk-action-matrix.md` 缺实际批准人、批准日期和证据链接，DEC-004 的 closed 记录不满足 `open-decisions.md` 自己的关闭条件。UX 不应把缺证据状态表现为正式批准。
5. **视觉假设标记不统一（低）**：校徽安全区仍用普通 `[ASSUMPTION]`，圆角/断点已使用 `[PRODUCT BASELINE v1.0 / DEC-*]`。虽然 Open Questions 引用了 DEC-008，但实现清单可能遗漏该假设。

## Epic 与 Story 质量审查

### 总体结构

8 个 Epic 均描述可感知的用户或运营结果，没有以数据库、API 或基础设施为目标的纯技术 Epic。顺序总体遵循“可信接入 → 数据质量 → 住宿闭环 → 多场景 → 协同 → 治理 → 移动 → 智能增强”。Story 1.1 和 2.8 是技术使能项，但分别绑定可重复工程基线和可验证连续性结果，可作为必要例外。

### 🔴 严重问题

1. **Story 3.6 前向依赖 Story 4.7**：AC-3.6.1 要求展示“合证优先级”，该事实直到 4.7 才计算。3.6 必须限定为展示当前已存在字段，真实合证投影移至 4.7；禁止用空值、样例值或未来 schema 证明完成。
2. **Story 3.10 前向依赖 Story 5.3**：AC-3.10.1 要求展示“协同结果”，但 TransferOrder/TransferEvent 到 Epic 5 才存在。3.10 只应验收机器信号、人工核实和关怀行动；5.3 增量验收协同结果时间线。
3. **Story 4.1 无法证明 FR21**：唯一 AC 只验证 `ECON-012 blocked` 时拒绝发布；approved 后仅列展示字段，没有输入批次、基线窗口、候选生成、质量门、排除、去重、时钟、重放和失败路径。该 Story 当前只能证明治理阻塞，不能证明经济压力场景功能。
4. **MVP 住宿闭环无可执行生产规则**：Story 3.4 要求 `ACC-SAFE-001/002 approved/production`，而规则目录仍为 review、阈值/生效日期待定。Story 正确表达了门禁，但 Epic 3 当前不能独立交付其用户价值；不得将其标为 ready。
5. **UX-only 合规材料工作流没有 Story**：上传、复核、退回、完成状态既没有 PRD FR，也没有架构 owner/状态机，更没有 Story。必须删出 UX 或完成正式需求变更后再拆 Story。

### 🟠 主要问题

1. **FR4 生产待办生命周期不完整**：1.9 只做适配器契约；2.5、3.4、5.1 只创建待办。没有 Story 完整验收临期、超期、升级一次性提醒，责任/截止/状态更新，拒绝/合并/撤权/终态关闭，以及消息失败恢复对账。FR4 的“覆盖”目前只是部分覆盖。
2. **FR40 通知要求未落入 Story 5.2**：PRD 明确临期、超期和退回通知责任人；AC-5.2.1 仅描述状态迁移和授权，没有通知、幂等或失败路径。
3. **Story 3.1 成功路径当前不可测试**：目录中没有 approved 且有生效日期的规则。AC 同时覆盖保存、测试、发布、回退，却没有可用夹具或已批准匿名样本包。只能验收基础设施和拒绝路径，不能完整关闭 FR14/FR15。
4. **三类规则状态混用**：Story 3.1/3.2 沿用 Architecture 与规则目录的不同状态词，没有规范治理审批状态、运行状态和质量可用状态的映射；测试无法唯一确定合法迁移和组合。
5. **Story 4.2、4.3 的 Given 当前不可满足且阻塞未显式化**：`NIGHT-001`、`ACADEMIC-001` 均为 review、无生效日期，AC 直接假定“运行/生效”。应像 3.4 一样声明 approved/production 与证据门禁，或标为 blocked backlog。
6. **Story 4.4 对 blocked 经济规则形成不必要耦合**：Given 要求经济、夜间、学业等源规则均交付。若专项可由至少一个已批准源组成，应定义 required/optional；若全部必需，应显式标注被 ECON-012 阻塞。
7. **FR32 关怀动作目录验收缺失**：PRD 要求受控动作目录、至少六类动作、每类完成事件与时限来源；AC-3.8.1 只覆盖三类核实结果和表单提交，无法证明动作目录完整性或合法闭环。
8. **FR56 配对规则覆盖不完整**：AC-4.8.1 未明确“重复刷卡先去重”；重复输入下统计结果不可判定。
9. **转介关闭 owner/顺序不明确**：Story 5.2 允许协同人员执行关闭；UX UJ-4 又让辅导员最终闭环后关闭工单；Story 5.3 仅说明工单不自动关闭线索。两个聚合各自关闭权限、前置条件与互不自动关闭规则仍不完整。
10. **多数 Story 把业务错误路径委托给通用 DoD**：2.1、2.2、3.11、3.12、4.2–4.6、6.1–6.4 等通常只有一个长 AC。通用“错误路径、幂等、审计”无法替代成功、拒绝、并发/重放、依赖降级的具体预期结果。
11. **Story 2.4 的规则依赖注册边界含混**：它在 3.1 创建规则生命周期前登记 ruleId/version。虽然规则目录可作先行契约，但 AC 未明确只允许引用目录中既有版本、不得创建 RuleVersion 运行实体，仍可能形成未来对象依赖。

### 🟡 次要问题

1. **Story 3.13 的兼容 schema 语义模糊**：允许预留协同字段，但必须把“未交付/不可用”与数值 0 区分，并用 schema 兼容测试证明；真实协同指标仅由 6.4 验收。
