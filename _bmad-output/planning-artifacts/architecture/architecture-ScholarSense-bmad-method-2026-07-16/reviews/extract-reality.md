# 既有技术现实提取（Brownfield / Reality Sweep）

## 1. 范围与判定口径

本记录只提取事实，不设计目标架构。扫描范围：

- `docs/input/原型/frontend/`（排除 `node_modules/`、`dist/`）；
- `docs/input/原型/frontend/package-lock.json`；
- `docs/input/文档/需求列表.md`；
- `extract-requirements.md`、`reconcile-prototype.md`；
- 最终 `prd.md`，仅用于判断早期技术约束是否仍绑定。

判定优先级为：最终 PRD 的已决策前提与验收要求 > 输入对账结论 > 早期需求清单 > 原型代码和 mock。原型可证明当前演示实现及信息架构，不能证明生产后端、集成、持久化、安全或运维能力已落地。

## 2. 已存在且可 ratify 的技术现实

### 2.1 可运行前端演示的工程基线

- 存在独立 Vue 单页应用：Vue 3、TypeScript、Vite、Vue Router、Pinia、Element Plus、ECharts / vue-echarts；使用 ESM、懒加载视图、`@` 路径别名和中文 Element Plus locale。
- `package-lock.json` 为 lockfile v3。当前锁定解析版本包括 Vue 3.5.38、Vite 5.4.21、TypeScript 5.9.3、Element Plus 2.14.2、ECharts 5.6.0、Pinia 2.3.1、Vue Router 4.6.4、Axios 1.18.1、vue-echarts 7.0.3。`package.json` 声明范围明显更早，因此“声明版本”与“实际 lock 解析版本”须分开记录。
- 已有页面级路由、管理端与驾驶舱两类布局、通用 UI 组件、图表封装、领域 TypeScript 类型、mock 数据分包、全局设计令牌和格式化/类型检查/构建脚本。这些可作为前端组织方式的既有证据。
- Vite 当前配置 `base: /scholarsense/`，本地 `/api` 代理到 `VITE_GATEWAY_URL`（默认 `127.0.0.1:8080`）；`.env.example` 声明 `VITE_API_BASE=/api` 与网关地址。它证明原型为网关接入预留了约定，不证明生产路径或网关实现已确定。
- 代码注释明确预期由统一门户承载，并为 qiankun / 无界生命周期留出接入点，但当前仍直接 `app.mount('#app')`，尚无微前端 bootstrap/mount/unmount 实现。

### 2.2 可 ratify 的产品—技术边界

- 最终 PRD 已绑定：本模块复用学校统一身份认证、统一门户、公共消息/待办/通用流程、权威组织与责任关系、校级日志监控配置和基础设施；不重建这些公共能力。
- 最终 PRD 已绑定：首期无学生端；首期智能主干为“规则 + 个人基线 + 结构化解释”，生成式能力后置且必须可降级到稳定规则基线。
- 最终 PRD 已绑定：组合授权必须覆盖角色、组织、责任学生和有效协同任务；字段展示、列表、详情、导出和接口返回保持同一授权/脱敏结果；运维默认无业务明细权限。
- 最终 PRD 已绑定：关键业务状态、规则/策略变更、对外集成和异步任务需具备幂等、重试、灰度、回退、审计和 traceId 关联能力；数据质量不达标必须在正式线索前熔断。
- 最终 PRD 已绑定：敏感字段按学校既定方案加密存储、密钥集中管理、全链路传输保护、审计防篡改且至少留存 6 个月；具体算法或产品未在最终 PRD 中锁定。

## 3. 仅为原型、不能 ratify 为生产现实的内容

- 全部业务数据来自 `src/mock/*`；扫描未发现实际 Axios/fetch 调用，Axios 仅列为依赖。没有后端源码、数据库迁移、API 契约、消息中间件、批处理、规则引擎或部署清单可供 ratify。
- 登录是 Pinia 内存状态中的“选择角色”，路由守卫仅检查 `currentUser !== null`；没有 CAS 跳转/回调、会话续期、内部令牌、服务端权限校验或组织责任关系同步。
- 菜单 `meta.roles`、`canViewSensitive`、`dataScope` 和前端脱敏函数只是演示可见性，不是安全边界。尤其协同角色可进入一人一档、领导 `dataScope: all` 均被最终 PRD 收紧。
- 保存、导出、审批、状态变化、熔断恢复、规则发布/回退、标签新增、审计写入、自然语言问数等大量动作仅显示消息或修改局部内存；不证明持久化、授权、后台任务、真实审计或模型能力存在。
- `/scholarsense/`、5173 端口、8080 本地网关、qiankun / 无界注释均为演示/接入预留，不是生产部署合同。
- 原型中的规则阈值、角色权限宽度、KPI、学院排名、算法指标、推送频率和 mock 状态不能提升为生产默认值。
- 需求清单称“M0 脚手架/CI/基础设施/安全基线已完成”，但本次可见材料只能证明前端原型脚手架；最终 PRD 也要求实施团队核验真实状态，故不得 ratify 为完整工程底座已完成。

## 5. 需要显式裁决或外部事实核验的冲突

1. **工程状态冲突**：早期清单称 M0 已完成；可见仓库只有纯前端原型，且最终 PRD 明示不得将其等同于完成。架构必须把“生产工程底座是否另存于其他仓库/平台”作为外部事实核验，未获证据前不能按 brownfield 后端 ratify。
2. **门户集成形态**：最终结果要求嵌入统一门户，但 qiankun、无界或其他宿主协议未定；原型的 base path 与注释不能代替宿主契约。
3. **身份传播机制**：早期要求网关签发短时内部令牌并贯通微服务，最终 PRD 未锁实现；需从学校身份/网关现实取得协议，否则不可把 JWT、session 或 token exchange 写成既定事实。
4. **服务范式**：早期措辞预设微服务、Nacos、Spring Cloud Alibaba；最终 PRD 只约束边界结果。是否继承学校既有 Java 微服务平台必须由真实部署/平台标准证明。
5. **安全技术细节**：国密与国际算法、数据库 TDE、字段级应用加密、密钥服务边界均存在早期并列写法；最终仅绑定“按学校既定方案”。架构不能自行选择后再标为 `[ADOPTED]`。
6. **通知与转介集成**：早期来源对站内、云中苏大 App、短信/企微等渠道，以及平台内工单或外部系统回填存在差异；最终 PRD绑定复用公共消息/待办/流程和既有系统接口，但具体通道与职责边界仍需公共能力契约核验。
7. **前端版本基线**：`package.json` 的 caret 范围与 lock 中实际解析版本跨越多个 minor 版本。若原型作为正式前端 seed，应明确以 lock 解析版本还是重新批准的受控版本为发布基线；本提取不作选择。

## 6. 给架构脊柱的事实边界

- 可将“Vue/TypeScript 前端原型及其页面组织”记为现有 seed 候选，但不能据此宣称生产前端已经接入门户或 API。
- 可将最终 PRD 的公共能力复用、安全/审计结果、质量熔断、状态一致性、韧性和性能指标作为绑定约束。
- 不可把微服务、Spring Cloud Alibaba、Nacos、短时令牌、qiankun/无界、SM4/AES/TDE、Prometheus/Grafana/SkyWalking写成已被现有代码或最终 PRD ratify 的技术现实；它们至多是需要校方平台证据确认的历史候选。
- 当前材料不足以 ratify 后端边界、数据所有权实现、存储产品、消息系统、规则执行引擎、部署拓扑或灾备实现。
