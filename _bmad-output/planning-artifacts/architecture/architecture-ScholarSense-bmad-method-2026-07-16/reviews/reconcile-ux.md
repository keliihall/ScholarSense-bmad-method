# UX → 架构脊柱对账

> 对账范围：最终 `DESIGN.md`、`EXPERIENCE.md`、UX `.memlog.md`、`reviews/extract-ux-architecture.md` 与当前 `ARCHITECTURE-SPINE.md`。本文仅报告未落地、冲突或静默丢失项，不修改架构脊柱。

## 结论

当前脊柱已完整承接主要结构性约束：双宿主形态、Vue/Element Plus/ECharts、唯一服务端状态、模块写所有权、命令幂等与乐观并发、Outbox、组合授权、敏感字段三态、不可变审计、数据质量熔断、规则版本、授权读模型、异步作业和 WCAG 2.2 AA。

仍有 **7 项 P0、10 项 P1、7 项 P2** 未明确落入 Rule/一致性约定/Deferred。未发现与 UX 双脊柱直接相反的硬冲突；主要风险是“能力映射或泛化表述存在，但关键体验语义静默丢失”，导致下游 Epic 可能作出互不兼容的实现。

优先级定义：

- **P0**：若不补入脊柱，下游将可能破坏业务正确性、安全性、时钟或真实闭环。
- **P1**：会造成跨 Epic 契约分歧、连续性或验收目标落空。
- **P2**：主要影响体验一致性或验收完整性，可由 UX 双脊柱直接约束实现，但当前脊柱未显式承接。

## 冲突核查

未发现需要判定为硬冲突的条目，但有两处需防止误读：

1. **“单一跨端前端”措辞比 UX 更强。** UX 明确绑定的是 Web 与 WebView 共用唯一业务状态和同一产品契约；AD-17 标题及能力映射写成“单一前端”。若其意图是强制单代码库/单构建产物，这是架构新增决策而非 UX 已确认事实；Rule 当前只写同一 API 和状态，尚未真正要求单制品。应澄清标题含义，避免把 UX 来源误标为已采纳。
2. **固定版本栈不是 UX 约束。** UX 只继承 Vue 3、Element Plus、ECharts，不确认具体版本。脊柱已说明前端版本来自 `package-lock.json`、后端版本为 `[ASSUMPTION]` seed，因此不构成冲突；定稿时仍应保留来源标签，不能称为 UX 已决定。

## 已完整落地，无需补写

- 模块化边界与唯一写所有权。
- 所有业务变更经应用命令、幂等键、聚合版本与状态事件。
- 证据来源、源版本、窗口、质量快照与截止时间固化。
- 数据质量熔断、校验、样本重算、授权恢复、恢复观察及不补发失效线索。
- 规则不可原位修改、受限声明式执行、候选去重、跨类别独立线索引用。
- Outbox 至少一次投递、消费去重、双方 ID 对账与外部失败真实状态。
- 每请求组合授权、即时撤权、路由非安全边界、不泄露对象存在性。
- 敏感字段明文/脱敏/不可见三态覆盖 API、缓存、导出与无障碍树。
- append-only 审计、traceId、至少 6 个月、防篡改归档。
- 可重建读模型、统计口径/周期/截止时间/质量状态、授权汇总问数。
- 持久后台作业与用户会话解耦。
- Web/WebView 服务端确认成功、跨端同步 P95≤5 秒、图表/表格共享 DTO。
- WCAG 2.2 AA 的总底线及表格、图表等价数据、错误摘要、焦点恢复、live region 引用。

## 建议的定稿处理顺序

1. 先处理 UX-R01—R07，将业务时钟、状态机、用途隔离和禁止下钻变为不可歧义的不变量。
2. 再处理 UX-R08—R17：能导致下游分歧者形成 Rule；校方契约未到位者进入 Deferred 并写清当前边界与重访条件。
3. UX-R18—R24 不必膨胀为独立 AD；可通过一致性约定明确“双脊柱为实现与验收权威”，并保留 `[ASSUMPTION]` 和未完成视觉 QA 的来源状态。
