# Story 1.5：提供授权审计检索、归档与销毁证明

Status: done

> 证据边界：本 Story 交付真实的、服务端逐请求授权的审计检索 API/UI，以及独立归档合同、legal hold/保留执行模型和可重放的审计域 `RetentionExecution`。因当前生产在线 ledger 是严格连续且禁止 `UPDATE/DELETE/TRUNCATE` 的单链，本 Story 只在隔离的合成/匿名 fixture scope 上演练到期销毁，不删除生产 `ao_audit_ledger`，也不签发、持久化或展示“实际跨域 `DeletionReceipt` 已完成”。全数据域消费者、读模型、对象、缓存、索引和备份水位全部确认后的真实 `DeletionReceipt` 唯一属于 Story 6.6。

## Story

作为具备当前审计检索权限的审计或平台运维人员，
我希望按用户、对象、动作、时间、结果和 `traceId` 检索不可变审计记录，并让受控执行器依据已批准保留计划完成可核验的归档与销毁演练，
以便证明审计事实仅在授权用途内可见、原始事实未被改写、legal hold 始终优先，而且域内执行证据不会被误当成全系统生产销毁回执。

## 业务与交付价值

- 完成 FR-8 的搜索合同、实现和 RFP conformance 证据，并以 Story 1.3 的原子事实和 Story 1.4 的连续哈希链为唯一事实来源；在 Story 1.6/1.7 提供权威角色上下文前，production runtime 只允许 fail-closed，不得宣称生产授权 happy path 已启用。
- 作为 NFR-15 的最终 Story，补上独立归档对象、manifest、回读完整性和可信时间证据；不把在线 hash 链夸大为 database owner 无法重写的不可否认证明。
- 作为 NFR-12 的审计记录域 contributor，固化 `RS-1.0.0`、3 年审计保留、targeted legal hold 和域内执行证据；Story 6.6 才验收全域销毁。
- 将 R3 的业务审计元数据用途与 R7 的技术审计元数据用途严格隔离，避免以“运维/审计”名义获得统一全量明文。

2. 范围内：只读审计检索 API/UI、稳定快照分页、R3/R7 用途隔离与字段投影、检索行为自身审计、审计搜索读模型、归档/保留/hold/执行证据合同、存储中立的独立归档端口、合成 fixture 销毁演练、真实 PostgreSQL 权限与故障证据。
3. 范围外：同步明细导出或直接文件下载、通用角色首页、全系统字段投影引擎、人工 archive/destroy 按钮、生产 ledger 行删除/分区退休、全域对象/缓存/索引/备份删除和实际 `DeletionReceipt`。导出由 Story 3.14a—c，通用身份/角色由 1.6a—1.8，真实销毁闭环由 6.6 交付。
4. RFP 仅定义 R3 `audit.search-business-metadata` 与 R7 `audit.search-technical-metadata`；不存在第八个“审计员”角色。未列 action 默认 deny，HRAP 也没有 `audit.archive` / `audit.destroy` / `retention.execute`，因此浏览器端不得出现手动归档或销毁命令。
5. 当前身份会话投影尚无权威 role/object model；不得把前端菜单、客户端 role、测试 header 或静态 allow 当作生产授权。为本 Story新增最窄的 identity-owned 公共授权端口和合规 fixture；生产绑定在权威角色不可用时必须拒绝全部审计结果。后续 1.6/1.7 可替换端口实现，不得让 `audit-operations` 读取 identity 表或导入其 internal 类型。
6. `audit-operations` 唯一拥有 ledger、审计搜索读模型、归档状态与 `RetentionExecution`；归档对象必须位于与在线 PostgreSQL ledger 故障域分离的受控存储边界。未选定校方对象存储供应商前，只冻结 portable port/conformance suite，禁止引入 AWS、MinIO 或其他厂商 SDK/类型并宣称生产 WORM 已配置。
7. RS-1.0.0 对安全与业务审计采用 3 年保留期，覆盖 PRD 的“至少 6 个月”下限。所有时间存 UTC，接口使用带 offset 的 ISO 8601；查询和保留窗口使用半开区间 `[startAt, endAt)`，`now == retentionEnd` 进入到期候选，hold 的 `startAt` 含、`endAt` 不含且结束后只重新评估。
8. 本 Story 的 approved Ready When 明确允许合成/匿名 fixture 验收，因此 `done` 可表示“搜索/归档/保留软件能力与 conformance 完成”，不表示生产角色源、校方 WORM 或真实跨域删除已启用。生产审计搜索 activation 必须在 Story 1.6/1.7 接入权威角色后另做 R3/R7 E2E 证据；在此之前 capability manifest 必须记录 `productionAuthorizationEnabled=false`。

## Acceptance Criteria

### AC-1.5-HAPPY：授权检索与审计域保留演练

**Given** Story 1.4 的 ledger/head 全链验证健康，`RFP-1.0.0`、`RS-1.0.0`、legal hold 规则、`DeletionReceipt` schema 和合成/匿名 fixture 均通过合同检查

**When** 具权用户执行只读检索，且受控 service identity 在 fixture scope 上执行归档与到期销毁演练

**Then** 用户只看到当前 action/scope 允许的字段投影，归档与保留执行准确引用 `RS-1.0.0`

**And** 系统生成可汇总、可重放、明确标注 `scopeType=audit-domain` 和 `nonProductionEvidence=true` 的 `RetentionExecution`，不生成或暗示实际跨域 `DeletionReceipt`。

### AC-1.5-SEARCH：组合筛选、稳定快照与只读语义

**Given** fixture 含不同用户、对象、动作、时间、结果和 `traceId` 的 ledger facts

**When** 分别或组合这些条件查询

**Then** 返回条件交集，按 `occurredAt DESC, ledgerSequence DESC` 稳定排序；首次响应冻结 `asOfSequence`，后续页必须携带同一值，期间新 append 不得造成重复、漏项或翻页漂移

**And** 响应携带 `sourceLedgerHead`、`projectionWatermark` 和 `dataCutoffAt`；首次快照只能冻结到 `min(sourceLedgerHead, projectionWatermark)`。若客户端请求的 asOfSequence 高于 projection watermark，则以 `AUDIT_SEARCH_PROJECTION_NOT_CAUGHT_UP` 拒绝而不返回 partial rows；只有完整覆盖该快照的空结果才能解释为“无记录”

**And** API 明确分页并限制 `size <= 100`；非法页码、size、时间区间、枚举、UUID/token/trace 格式返回标准错误 envelope，而任何检索路径都不能更新、删除或修订 ledger 原始记录。

### AC-1.5-AUTH-DENIAL：R3/R7 用途隔离与无泄露拒绝

**Given** R3、R7、多角色用户和其他无审计 action 的用户请求相同记录、对象或 `traceId`

**When** 服务端逐请求重检当前会话、`RFP-1.0.0`、action、scope 和字段投影

**Then** R3 仅能调用 `audit.search-business-metadata` 并获得 R3 的 B/I/C/S/E/N/G/T 投影；R7 仅能调用 `audit.search-technical-metadata` 并获得 R7 投影；多角色对同一对象/action 使用 deny-overrides 和最严格适用字段投影，其他角色默认拒绝

**And** 拒绝与不存在使用同形错误 envelope，不泄露 auditId、对象是否存在、结果数量、隐藏字段键/长度、archive 或 hold 状态；前端隐藏入口只改善体验，不能充当授权证据。

### AC-1.5-AUTH-BINDING：conformance 完成但生产默认关闭

**Given** RFP-1.0.0 fixture 可以提供 R3/R7 权威 decision，而当前 production session 尚无 Story 1.6/1.7 的权威 role/scope context

**When** 分别运行 conformance profile 与 production profile

**Then** conformance profile 完整通过 R3/R7 allow/deny/projection oracle；production profile 在权威 adapter 未绑定时对所有 search/evidence read 返回统一 fail-closed 错误，不能用 fixture、header、静态配置或客户端 role 解锁

**And** Story 完成记录明确区分 `conformanceVerified=true` 与 `productionAuthorizationEnabled=false`；后续 1.6/1.7 只有通过真实身份、撤权和下一请求重检 E2E 后才能翻转 activation 状态。

### AC-1.5-TOKENIZATION：可轮换搜索令牌与输入最小化

**Given** ledger 只保存 `actor/object/sourceIp/aggregate/grant` 的搜索令牌、profileVersion 与 keyVersion，不保存可搜索明文

**When** 用户按原始用户/对象标识筛选

**Then** audit application 仅经 identity-access 的公开 tokenization/query 端口获得该用途下所有仍在保留窗口内的受支持 keyVersion token，按参数化 SQL 查询；不得复制 HMAC 实现、读取密钥、只查当前 key 或把原始输入写入 ledger/日志/指标

**And** KMS、token profile、历史 key 清单或 authorization port 缺失/漂移时 fail closed；原始敏感筛选值不进入 URL、浏览器持久存储、telemetry、错误 envelope 或检索自身的 audit payload，仅记录筛选类别和 canonical filter digest。

### AC-1.5-REQUEST-SECURITY：POST 查询的 CSRF、来源与缓存保护

**Given** search 使用 POST body 承载敏感筛选值

**When** 浏览器提交 `POST /api/v1/audit-records/search`

**Then** client 先经现有 identity CSRF proof 流程取得证明并发送约定 header，服务端同时校验 CSRF、Origin/Referer、认证 cookie 和 content type；缺失、跨源、过期或重放请求以无泄露 403 拒绝并审计

**And** search 与 execution evidence 响应均设置 `Cache-Control: no-store`、`Referrer-Policy: no-referrer`；executionId 是无业务语义的 UUIDv7，不能编码主体、scope 或时间。

### AC-1.5-READ-AUDIT：检索行为自身先审计后返回

**Given** 审计检索属于敏感读取

**When** 查询即将返回成功结果或授权拒绝

**Then** `audit-operations` 按 Story 1.3 的公共 `LocalAuditFact`/outbox 合同，在返回敏感内容之前原子提交模块本地 search audit fact；提交失败时零结果返回

**And** 本次查询先冻结 `asOfSequence`，防止 search fact 递归进入当前结果；该事实归集后可被下一次具权查询检索，自动后台 projection/归档处理不得自审计形成无限循环。

### AC-1.5-ARCHIVE-INTEGRITY：独立归档可回读验证

**Given** 待归档 ledger 区间连续、已完成全链验证且没有 unresolved permanent finding

**When** 归档执行器经 storage-neutral `AuditArchivePort` 写入独立 append-only/object-lock 边界

**Then** archive manifest 至少绑定 schemaVersion、scheduleVersion、scope/fixture hash、sequence 起止、recordCount、firstPreviousHash、lastEntryHash、archive object/version id、content digest、创建主体、可信时间和 `traceId`

**And** 成功前必须从独立边界回读并重算 record/count/hash/digest；相同 object/version 不得覆盖，保留期内不得删除或缩短 retention，legal hold 可独立施加；归档成功永远以新事实表示，不修改 ledger 行。

### AC-1.5-RETENTION-BOUNDARY：三年保留与 targeted legal hold

**Given** RS-1.0.0 的审计保留为 3 年，fixture 覆盖到期前、恰到期、到期后及 legal hold 的 start/end/reviewAt 边界

**When** 执行器使用可信服务端时间和冻结 scope watermark 评估资格

**Then** 未到期事实不得销毁，`now == retentionEnd` 才成为候选；命中有效 hold 的事实进入 `blocked-by-legal-hold`，未命中事实不被整域阻塞

**And** hold 至少绑定 purpose、scope、authority、startAt、endAt、reviewAt 和版本；hold end 只触发完整重新评估，不得绕过归档回读、ledger health、policy 或 watermark 守卫直接销毁。

### AC-1.5-ILLEGAL-STATE：政策、完整性或作用域非法时失败关闭

**Given** schedule 缺失/未批准/漂移、hold 字段不完整、scope 非法、ledger/head unhealthy、待归档范围未全链验证、trusted time 过期，或 archive manifest 不匹配

**When** 尝试归档、销毁演练或把 execution 标为成功

**Then** 不修改 ledger、archive 对象或成功证据，`RetentionExecution` 保持 `blocked` 或 `failed` 并记录稳定 error code/guard list/traceId

**And** 不创建成功 receipt、不自动修链、不以数据库 owner 权限绕过 writer/verifier 分离，也不将技术错误降级为可见的敏感搜索结果。

### AC-1.5-DEPENDENCY-FAILURE：依赖故障无假成功

**Given** PostgreSQL、authorization、KMS/key catalog、trusted time、独立 archive storage、回读校验或本地审计提交任一不可用

**When** 检索、归档或销毁 fixture 运行

**Then** 敏感搜索与销毁步骤 fail closed；已确认的独立归档步骤保留为可重试事实，但整体不得显示 succeeded

**And** 未确认对象、版本和 watermarks 显式列入 execution，销毁绝不先于 archive read-after-write verification 与执行事实持久化。

### AC-1.5-IDEMPOTENCY：执行重放与冲突

**Given** 相同 `retentionExecutionId + scheduleVersion + scopeHash + asOfSequence` 被重放

**When** worker 重试、重启或重复接收

**Then** 返回相同逐步骤结果和 evidence references，不重复创建 archive object、不重复销毁 fixture、不重复 append execution fact

**And** 同 executionId 携带不同 schedule、scope、watermark 或请求 digest 时稳定冲突并保留原结果，不以 last-write-wins 覆盖。

### AC-1.5-CONCURRENCY：hold、策略与 watermark 竞态

**Given** 资格预览后新增/更新 hold，或 scheduleVersion、scope version、ledger head、object version、selection watermark 发生变化

**When** 旧 worker 提交 archive/destroy/succeeded 结果

**Then** 以 aggregateVersion/fencing 拒绝旧提交并重新计算，新增 hold 优先

**And** 不形成混合 policy、scope、object version 或 watermark 的成功 `RetentionExecution`；并发 worker 只能有一个胜者。

### AC-1.5-EVIDENCE-BOUNDARY：域内证据不冒充跨域回执

**Given** fixture 的归档与销毁演练全部通过

**When** 汇总 `RetentionExecution`

**Then** 它保存 `scopeType=audit-domain`、`fixtureId`、`nonProductionEvidence=true`、scheduleVersion、scope/hash、动作/状态、watermarks、archive digest、主体、可信时间、traceId 和 unmet guards

**And** runtime/schema/UI 均拒绝将其标记为 production cross-domain `DeletionReceipt`；本 Story 只能用 approved `DeletionReceipt` schema 的正反 fixture 做兼容检查，签发能力不可达且由测试证明。

### AC-1.5-NO-SYNC-EXPORT：导出边界

**Given** 用户位于审计搜索结果页

**When** 请求同步下载、浏览器拼装文件或绕过导出治理

**Then** 本 Story 不提供同步明细导出、CSV/Excel/PDF 下载或直接 archive object 下载

**And** 后续带出必须进入 Story 3.14a—c 的异步作业、字段投影、D3/D2 门禁、有效期、下载重检和全程审计。

### AC-1.5-UX：可恢复、可访问且不泄露的搜索体验

**Given** 具权用户通过直接深链进入审计检索

**When** 页面处于 loading、empty、filtered-empty、forbidden、error、degraded 或分页状态

**Then** 每种状态具有独立文案和唯一可执行恢复动作；动态结果/错误使用 live region，筛选条可清除，分页不是无限滚动，表格及控件可键盘操作，焦点可见，200% 缩放不丢操作，1366×768 无页面横向滚动，状态不只靠颜色表达

**And** 页面只呈现投影后的字段，不把隐藏字段放进 DOM/无障碍树；非敏感 action/outcome/time/page 可恢复到 URL，用户/对象原始筛选只存当前进程内存并在刷新、退出、会话失效或账号切换时清除

**And** continuation 只保存 `audit.search` route id，不保存 filters/executionId；重认证后回到审计页并重新鉴权，若 production 权威角色仍不可用则进入统一 forbidden 状态。

### AC-1.5-VERIFICATION：自动化和发布证据完整

**Given** 候选实现改变 Java、SQL、Vue、contracts、scripts 或发布受控字节

**When** Story 申请进入 `review`

**Then** 单元/应用/架构/前端测试覆盖授权矩阵、字段投影、组合查询、稳定快照、先审计后返回、token key rotation、三年/hold 边界、归档回读/tamper、幂等、并发、故障和证据边界

**And** 真实 PostgreSQL 18.4 clean + V000001→…→V000005 升级验证搜索索引、读角色、执行事实、无跨 schema FK，以及 writer/search/verifier 均不能 `UPDATE/DELETE/TRUNCATE ao_audit_ledger`；完整 backend/frontend/contract/release gate 和两次 clean-root 可复现构建通过。

## Dev Notes

### 不可协商的实施决策

- 搜索事实源始终是 Story 1.4 ledger；`ao_audit_search_projection` 是可重建 read model。禁止直接向 HTTP 返回 `payload` JSON、对 payload 做任意 JSONPath 搜索，或让 UI/SQL自行决定字段可见性。
- 授权必须有两层：HTTP adapter 的当前身份/request gate，以及 application 的 RFP action/scope/projection decision。后端每次请求重新计算；“有登录态”不等于有审计权限。
- 当前 `CurrentSessionProjection` 没有 roles，`AuthorizationRecalculationPort` 仍为 Story 1.2 临时 allow。实现者不得沿用它授权 audit search；权威绑定未就绪时的正确行为是 deny，而不是创建假角色表绕开 1.6/1.7。
- R3/R7 是两个用途视图，不是一个全量页面的前端遮罩。R3 的 `audit.search-business-metadata` 和 R7 的 `audit.search-technical-metadata` 必须在服务端选择不同投影；多角色按同一对象/action 最严格适用投影。
- 搜索 token 已在 `LocalAuditFact` 中携带 `tokenizationProfileVersion` 与 `keyVersion`。按原始用户/对象筛选必须覆盖仍保留记录使用的历史 key；只以当前 key token 查询会造成静默漏数。
- 检索自身是一条新的敏感读取事实。先固定 ledger head 为 `asOfSequence`，执行查询，原子保存本地 audit fact/outbox，再返回映射后的结果；不要让自身审计改变当前页。
- 在线 ledger 严格只追加且 verifier 依赖连续 sequence/hash。Story 1.5 绝不能赋予 retention worker 删除在线 ledger 的权限；生产分区退休/anchor/genesis/verifier 跨归档段协议尚未冻结，必须留给后续受控设计。
- 归档 port 表达能力，不表达厂商。只有 deployment 提供已批准、与 PostgreSQL 独立故障域、可证明 object-version/retention/hold/readback 的 adapter 时才能启用 production archive worker；fixture adapter 只能在 test profile 使用。
- legal hold 是 targeted pause，不是永久修改数据；hold 结束需要重新评估。任何 schedule/hold/scope/watermark 漂移均不得把旧 execution 标为成功。
- `RetentionExecution` 是域内逐步事实，不是退出证明。Story 6.6 才能在所有 owner/consumer/对象/备份确认后签发 `DeletionReceipt`。

### 建议 API 与数据合同

冻结 `POST /api/v1/audit-records/search` 为只读、`Cache-Control: no-store` 的查询合同，避免把用户/对象原始标识放入 URL：

- `view=business|technical`：必须与服务端允许 action 匹配，不能由客户端自由升级。
- `actorRef`、`objectType`、`objectRef`：原始 ref 仅置于受 CSRF 保护的 JSON body 并进入 identity token port，不得进入 URL、浏览器持久存储、日志或 telemetry。
- `action`、`occurredFrom`、`occurredTo`、`outcome`、`traceId`、`page`、`size`、`asOfSequence`：明确格式、边界和默认值。
- response：`items`、`page`、`size`、`total`、`asOfSequence`、`sourceLedgerHead`、`projectionWatermark`、`dataCutoffAt`、`retentionScheduleVersion`、`roleFieldPolicyVersion`、`projectionStatus`；items 只能是 RFP 投影 DTO，不能复用 `LedgerEntry`。`projectionWatermark < sourceLedgerHead` 可显示 degraded，但 asOfSequence 不得超过已完整投影的 watermark；客户端请求更高快照时不返回 partial rows。
- errors：沿用 `{code, message, traceId, fieldErrors}`；对 forbidden/not-found 采用同形、无对象细节响应。

search client 必须先获取现有 identity CSRF proof 并发送约定 header；服务端校验 CSRF + Origin/Referer + cookie + JSON content type。另冻结 `GET /api/v1/audit-retention-executions/{executionId}` 为只读执行证据查询；executionId 为 opaque UUIDv7，它不提供 start/cancel/delete/archive/download 命令，不返回 archive object 直链。两个端点均返回 no-store/no-referrer headers并写入 OpenAPI 3.1.2、JSON Schema 2020-12 与 golden tests；不要并存第二套漂移语义。

### 字段与隐私约束

- 按 `RFP-1.0.0` 的 B/I/C/S/E/N/G/T 字段 profile 输出；隐藏字段必须在 DTO 构造前移除，不能以 `null`、空串、占位符或 CSS 隐藏泄露字段键。
- v1 搜索/执行证据 DTO 冻结以下最小字段 oracle；任何新增字段必须先分类并扩展合同/正反 fixture，不能由 serializer 自动带出：

| 字段类 | v1 批准输出 | R3 business view | R7 technical view |
|---|---|---|---|
| B | `recordId`（opaque UUIDv7）、`ledgerSequence`、`occurredAt`、`outcome`、schema/policy/schedule version | C | C |
| I | `actorDisplayRef`、`objectDisplayRef`；M 值是请求内固定格式别名，不含 token 子串且不跨会话持久化 | M | H |
| C / S / E / N | 无；联系方式、关怀属性、证据正文、自由文本不属于审计 metadata DTO | H | H |
| G | `businessActionCategory`、`businessObjectCategory`、`rolePackageSummary`、`projectionScope` | C | M |
| T | `producerModule`、`eventType`、`reasonCode`、`traceId`、integrity/archive/projection status、archive digest/version reference、`sourceNetworkRecorded` | M | C |
| 全局禁止 | ledger `payload`、所有 `*SearchToken`、原始/完整 source IP、grant/aggregate token、密钥/version inventory、archive object URL、业务正文 | H | H |

- C 也只表示“批准最小字段可见”，不是回显底层存储。M 使用与原值长度无关的固定替代；I 的请求内别名只能帮助当前结果分组，不能成为长期关联标识。R7 的 `sourceNetworkRecorded` 只证明来源信息已按 FR-8 记录，不反解/回显原始 IP；原始 fact 与授权输出是两个不同合同。
- source IP 属技术元数据，不能因为 FR-8 要求“记录”就向所有审计用途展示明文。审计存储与授权输出是两个不同合同。
- telemetry 仅允许低基数 action/view/outcome/error code、latency、result bucket；禁止 actor/object token、raw filter、auditId、traceId 作为 metric label。结构化日志也不得包含学生标识、完整 token/payload 或 archive 内容。
- 页面设置 `Cache-Control: no-store`，不得缓存审计响应；浏览器进程内敏感状态在身份边界变化时清空。

- 保持模块化单体和六边形依赖 `domain ← application ← adapters`；跨模块仅用 `.api` 或版本化事件，禁止跨 schema SQL/FK。

### 测试要求

最低测试矩阵：

1. 授权：R3、R7、R3+R7、其他角色、撤权、过期 session、policy unavailable；逐字段断言 clear/masked/hidden，hidden key 不出现在 JSON/DOM/无障碍树。
2. 请求安全：合法 CSRF proof、缺失/错误/过期/重放 proof、同源与跨源 Origin/Referer、非 JSON content type、cookie 失效；403 envelope/审计无对象泄露，响应 no-store/no-referrer。
3. 搜索：actor/object/action/time/outcome/trace 单项与交集；UTC offset、`[start,end)`、invalid/reversed range、空结果、filtered empty、稳定 sort、第一页/末页/超界页、size 0/100/101；watermark 落后时仅冻结完整快照，过高 asOf 拒绝且不把 incomplete empty 解释为无记录。
4. 快照：搜索前/查询中/翻页时并发 append，证明固定 `asOfSequence` 无重复漏项；本次 search audit 只在后续查询可见。
5. token：Unicode NFKC、不同 domain、current/historical key、缺历史 key、KMS failure、token collision 负例；日志/URL/storage 扫描无 raw refs。
6. archive：连续范围、空范围、gap/head/hash 不健康、readback digest/count/hash、对象篡改、覆盖/提前删/缩短 retention 被拒绝、hold 阻断。
7. retention：3 年前/恰等于/后；hold start 前/等于/期间/end 等于/后、reviewAt、scope 命中与不命中；schedule/hold/watermark 并发变化。
