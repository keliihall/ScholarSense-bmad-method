# Story 1.1b：建立模块化生产工程骨架

Status: done

> 状态说明：本文已完成开发上下文，不表示执行依赖已满足。Story 1.1a 当前为 `in-progress`、尚未 `done`，且三份必需交接产物尚不存在；实施 1.1b 前必须先通过本文的“实施前置门槛”。

## Story

作为实施团队，
我希望获得一套可重复构建、可自动检查边界的生产工程骨架，
以便后续 Story 在统一的模块化单体与六边形架构上增量实现，不复制原型的演示假设或重复发明工程边界。

## 业务与交付价值

- 本 Story 是 Epic 1 的工程 enabler，覆盖 M0、AD-1、AD-2 和 AD-28；不是任何 FR/NFR 的最终 owner。
- 它把“只有规划基线和 Vite 5 演示原型”的现状转化为后续 Epic 可扩展的生产源码边界，重点防止反向依赖、跨模块内部访问、多重写所有权、配置串环境和原型污染。
- 交付责任链固定为 `1.1a 资产审计 → 1.1b 工程骨架 → 1.1c 生产前端/Performance Profile ADR → 1.1d CI 与供应链质量门`。

## 实施前置门槛

开始修改生产工程前，必须同时满足：

1. `sprint-status.yaml` 中 1.1a 已为 `done`，且其 Dev Agent Record/File List 反映实际交付。
2. 以下产物存在并通过 1.1a 定义的 schema、重放和漂移校验：
   - `_bmad-output/implementation-artifacts/1-1a-production-asset-inventory.md`
   - `_bmad-output/implementation-artifacts/1-1a-production-asset-inventory.schema.json`
   - `_bmad-output/implementation-artifacts/1-1a-prototype-pab-gap.md`
3. 所有 owner 为 1.1b 的 `migrate` / `replace` / `unknown-blocked` 项均有明确处置；未解决项不得被静默复制到生产树。
4. G-01/PAB-1.0.0 仍为 `approved-for-implementation`，且实施环境显式使用 JDK 25、Node 24.18.0 和 npm 11.16.0；不得用本机当前 JDK 26/Node 23/npm 10 的结果冒充 PAB 证据。

任一条不满足时，实施代理必须停止创建生产骨架，记录缺口并先完成 1.1a；不得用本 Story 文档的 `ready-for-dev` 代替执行依赖。

## Acceptance Criteria

### AC-1.1b-HAPPY：交付可重复验证的生产骨架

**Given** 实施前置门槛已全部通过，且 1.1a 的处置结论可追溯  
**When** 实施团队在清洁工作区按 README 执行 bootstrap、构建和验证  
**Then** 建立 `backend/`、`frontend/`、`contracts/`、`deploy/base/` 四个生产根边界，后端能在 JDK 25/Spring Boot 4.1.0 下完成编译、单元测试、架构测试与启动 smoke  
**And** `frontend/` 仅建立 `app/domains/components/shared` 及配置边界契约，不提前冻结 1.1c 所有的生产 `package-lock.json` 或兼容矩阵  
**And** 同一受控输入的两次清理后验证均成功，结构清单和规范化验证结果一致，不依赖原型 `node_modules`/`dist`、本机绝对路径或未记录的全局工具。

### AC-1.1b-ARCHITECTURE：自动守卫六边形依赖与写所有权

**Given** `shared` 与九个 AD-2 业务模块已建立  
**When** 执行源码与编译产物的架构守卫测试  
**Then** 每个业务模块固定为 `domain ← application ← adapters`，入站 `api` 只调用 application；domain 不依赖 Spring/JPA/HTTP/数据库或任何 adapter  
**And** 跨模块同步依赖仅可指向 `<module>.api`，禁止内部包导入、循环依赖、跨模块表访问和跨模块外键  
**And** `shared` 仅容纳 ID、时间、稳定错误、trace 和 outbox 等技术内核，不容纳业务聚合、授权策略或跨模块“便利服务”  
**And** 故意违规的 fixture 必须以非零结果失败，并报告违规包、依赖方向或 owner。

### AC-1.1b-MIGRATION：建立按模块归属的迁移种子

**Given** PostgreSQL 18.4 为 PAB 数据库基线，且各事实的唯一 owner 已由 AD-2 冻结  
**When** 检查后端迁移目录与模块注册  
**Then** 每个迁移都位于其 owner 模块的受控目录，命名、顺序和 schema/表前缀可机器检查  
**And** 边界测试拒绝跨 owner 的 DDL/外键或同一事实多重 owner 登记  
**And** 本 Story 不一次性创建全部业务表；无实际业务对象时只创建目录、所有权清单和测试 fixture，未来 DDL 由首次交付该能力的 Story 提交。

### AC-1.1b-FRONTEND：固化前端业务域与原型边界

**Given** Vite 5 原型被 AD-28 定义为迁移输入，不是生产基线  
**When** 建立前端目录、集中路由/theme/config 扩展位与边界规则  
**Then** `frontend/src` 至少含 `app/`、`domains/`、`components/`、`shared/`，业务域目录使用 kebab-case，Vue 组件使用 PascalCase  
**And** 不复制原型 `dist`、`node_modules`、mock 登录/用户、Pinia 客户端授权、路由角色判断、`127.0.0.1:8080` 默认网关或“微前端/各微服务”假设  
**And** 不建立独立移动底部导航、Toast 假闭环、Service Worker 或持久化业务缓存  
**And** 边界检查拒绝跨 domain 内部导入、持久化 Pinia/localStorage/IndexedDB/Cache Storage 中的业务对象以及未在 client allowlist 中的环境变量。

### AC-1.1b-CONFIG-RUNTIME：配置失效可见且环境/进程角色隔离

**Given** dev/test/stage/prod 与 `web-api`/`worker` 是受控环境和运行角色  
**When** 分别执行配置校验和启动 smoke  
**Then** 非密配置仅由环境注入或校级配置端口提供，密钥仅传引用；仓库、前端 bundle、镜像种子、日志和错误不含密码、token、私钥或学生明文  
**And** 各环境的账户、数据库、密钥引用、存储命名空间和外部端点不得串用  
**And** `web-api` 与 `worker` 使用同一后端制品角色契约；worker 不暴露业务 HTTP 入口，两者均只保留后续部署 Story 可延伸的健康探针种子  
**And** 必需配置缺失、类型非法、未知 profile/角色或跨环境组合时 fail fast，返回可定位但不泄密的稳定错误。

### AC-1.1b-CROSSCUTTING：用测试契约固化通用命令安全语义

**Given** 本 Story 不实现任何真实业务 FR，但原始 AC 要求骨架可验证授权、审计、幂等与 aggregateVersion 语义  
**When** 在 test-only 示例聚合/用例上执行表驱动契约测试  
**Then** 授权拒绝、依赖失败或审计提交失败均不改变状态，失败结果稳定可见且含 traceId  
**And** 同 `Idempotency-Key`+同请求哈希重放原结果且无重复副作用，同 key 异哈希返回 409/`*_IDEMPOTENCY_MISMATCH`  
**And** 过期 aggregateVersion 返回 409/`*_VERSION_CONFLICT`，含当前版本、最新操作者和时间；并发竞争中至多一个请求推进状态  
**And** 该 fixture 和契约不出现在生产 API/数据库，不使用 no-op 适配器、mock 登录或假成功响应冒充 1.2/1.3/1.7 及后续集成 Story 的真实能力。

### AC-1.1b-EVIDENCE-BOUNDARY：产生真实骨架证据且不越界

**Given** 1.1c 和 1.1d 分别拥有生产前端/Performance Profile ADR 与 CI/供应链运行证据  
**When** 归档 1.1b 的构建和测试结果  
**Then** 交付记录包含实际工具/JDK/Node 版本、精确命令、退出码、测试报告、受控输入摘要、输出结构清单和未决项  
**And** 本 Story 不声称浏览器/WebView、Performance Profile、视觉/WCAG 完整验收、离线同摘要制品、SBOM、provenance、签名、漏洞/许可证、生产 lock 批准或制品提升已通过  
**And** 上述未完成项均登记到唯一 owner Story 1.1c 或 1.1d；静态 Gate 批准不得写成运行测试通过。

## Dev Notes

### 当前仓库与代码事实

- 当前工作区不是 Git 仓库，无法提供最近 commit、远端、分支或 provenance 起点；本 Story 不得伪造 Git 情报。
- 当前没有生产 `backend/`、`frontend/`、`contracts/`、`deploy/`、后端构建、数据库迁移、CI 或产品测试；本 Story 预期以 NEW 文件为主。
- 唯一可见产品代码是 `docs/input/原型/frontend`：其 package 范围、lock 解析和行为均不是生产真实性。实际包含 mock 用户/登录、客户端角色守卫、本机网关默认值及微前端/微服务注释，均不得直接继承。
- 1.1a 的三份交接产物当前全部缺失；本 Story 的文档已就绪不能改变该执行事实。

### 架构符合性

- Java 根包为 `cn.edu.suda.scholarsense`；模块的文件夹/外部名使用 kebab-case，Java 包使用无连字符的小写单数形式。
- 命令为 `<Verb><Aggregate>Command`，查询为 `<Get|List|Search><Subject>Query`，端口为 `<Capability>Port`，领域事件为过去式 `<Aggregate><Fact>`，传输事件为 `<module>.<aggregate>.<fact>.v<major>`。
- 同步外部 API 使用 `/api/v1/...`；错误 envelope 为 `code/message/traceId/fieldErrors[]`；ID 为 UUIDv7，存储时间为 UTC，输出 ISO 8601 offset。
