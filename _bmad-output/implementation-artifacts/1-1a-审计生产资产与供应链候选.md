# Story 1.1a：审计生产资产与供应链候选

Status: done

## Story

作为实施团队，
我希望识别可复用生产资产、供应链候选和原型边界，
以便后续 Story 在可追溯的事实上建立生产工程骨架，避免把 Vite 5 原型或未验证依赖误当为生产基线。

## 业务与交付价值

- 这是 Epic 1 和全项目的首个实施 Story，无前置 Story，直接使用已批准的 `AUTH-2026-07-17-001` 与 `PAB-1.0.0`。
- 本 Story 交付“已核验的资产事实与处置建议”，为 1.1b 建工程骨架、1.1c 冻结精确 lock/性能 Profile ADR、1.1d 生成 CI/SBOM/provenance/签名证据提供输入。
- 本 Story 不实现生产应用、不创建生产工程骨架、不升级原型依赖、不声称已通过供应链或性能验收。

## Acceptance Criteria

### AC-1.1a-HAPPY：形成可追溯的资产清单与原型边界

**Given** `AUTH-2026-07-17-001` / `PAB-1.0.0` 已批准，且本工作区、原型、规划制品和可见制品来源可读  
**When** 实施团队执行资产与供应链候选审计  
**Then** 在 `_bmad-output/implementation-artifacts/1-1a-production-asset-inventory.md` 生成可机器检查的资产清单，每项至少包含 `assetId`、类型、实际路径/来源、owner（已知值或 `UNKNOWN`）、当前版本/摘要（可获取时）、生效/核验日期、证据链接、分类和处置结论  
**And** 分类必须从 `reuse-as-is`、`migrate`、`reference-only`、`replace`、`unknown-blocked` 中选择，每个非 `reuse-as-is` 项目都有理由、风险和后续 owner Story  
**And** 清单显式记录当前可见事实：工作区不是 Git 仓库；未发现后端、CI 或部署工程；`docs/input/原型/frontend` 是 Vite 5 / Vue 3 原型，包含源码、`dist` 与 `node_modules`  
**And** 同时生成 `_bmad-output/implementation-artifacts/1-1a-production-asset-inventory.schema.json`，并以自动化校验证明清单字段完整、`assetId` 唯一、证据路径可解析  
**And** 任何读取失败、无法确定 owner/版本或无法解析来源的项目都显式标记为 `unknown-blocked`，不得静默遗漏或推断为可复用。

### AC-1.1a-BASELINE：对账 PAB 与原型依赖

**Given** 原型 `package.json` / `package-lock.json` 和已批准的 PAB-1.0.0 可读  
**When** 审计工具生成依赖差异  
**Then** 在 `_bmad-output/implementation-artifacts/1-1a-prototype-pab-gap.md` 逐项列出原型声明范围、lockfile 解析版本、PAB 精确基线、差异类型、迁移风险和 owner Story  
**And** 至少覆盖 Node/npm、Vite、Vue、TypeScript、vue-tsc、Vue Router、Pinia、Element Plus、ECharts/vue-echarts、测试工具、`axios`、`@vitejs/plugin-vue`、`@element-plus/icons-vue` 和 `@types/node`  
**And** Vite 5、TypeScript 5.x、Pinia 2、ECharts 5 及其组合必须标记为 `reference-only`/`migrate`，不得标记为生产基线  
**And** PAB 中未冻结的原型依赖不得自动进入生产候选；必须给出“移除、替换或在 1.1c 中通过新 ADR 批准”的显式处置  
**And** 报告明确区分“本 Story 核验的当前事实”与“1.1c/1.1d 尚未产生的 lock、兼容性、SBOM、provenance、签名、漏洞/许可证及浏览器/WebView 证据”。

### AC-1.1a-REPRODUCIBLE：审计结果可重复且可检测漂移

**Given** 完全相同的工作区输入和审计工具版本  
**When** 两次执行审计  
**Then** 除 `generatedAt`/核验时间外，规范化后的资产顺序、标识、摘要、分类和差异结果一致  
**And** 审计必须忽略时间戳、缓存、`.DS_Store`、`node_modules` 内部文件和已生成 `dist` 文件的不稳定元数据，但仍须把 `node_modules`/`dist` 作为顶层资产与污染/制品风险记录  
**And** 更改任一受控输入后，校验能识别漂移并以非零退出码失败，且不覆盖上一份已批准证据  
**And** 自动化测试覆盖正常盘点、缺失路径/无权读取、缺失 owner/版本、断链证据、重复 `assetId`、同输入重放和受控输入漂移。
