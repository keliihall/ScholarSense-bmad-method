# Story 1.4：归集不可变审计账本并检测篡改

Status: done

> 证据边界：Story 1.3 已交付模块本地 `LocalAuditFact`、完整 outbox、可信时间绑定和 PostgreSQL 原子提交证据。本 Story 只交付集中式、仅追加、可检测篡改的在线 `AuditLedger`，以及归集、重放、积压安全门和完整性告警。它能发现缺行、改行、head 漂移和幂等碰撞，但不能抵御能够重写整条链并重算哈希的数据库 owner/superuser；校方 WORM 归档、授权检索和归档回执属于 Story 1.5，全域销毁证明属于 Story 6.6。不得把本 Story 宣称为 FR-8、NFR-15 或不可否认性的最终闭环。

## Story

作为审计运维人员，
我希望将各模块本地审计事实幂等归集到具有连续序号和哈希链的集中账本，并及时发现缺口、重复、篡改与积压，
以便我能证明在线审计链的完整性，并在证据不可信时阻止高风险读写继续扩大风险。

## 业务与交付价值

- 把 Story 1.3 的模块本地原子事实转化为可集中核验的在线证据链，为 FR-8 提供第二个 phase/enabler。
- 通过严格连续的 `ledgerSequence`、`previousHash` 和可重算的 `entryHash` 提供 NFR-15 的在线防篡改证据，但不替代 Story 1.5 的独立归档/WORM 证据。
- 通过 `traceId`、稳定 finding code、归集水位与积压状态支撑 NFR-24 的跨 HTTP/worker/outbox 追踪，同时不泄露学生明文、证据正文或高基数标签。
- 将“积压到什么程度必须拒绝高风险读写”冻结为版本化、可测试的安全政策，关闭实施准备度报告中 Story 1.4 的未决项。

## Acceptance Criteria

### AC-1.4-HAPPY：幂等归集并形成连续哈希链

**Given** identity-access 中存在符合 `LOCAL-AUDIT-OUTBOX-1.0.0` / `LOCAL-AUDIT-FACT-1.0.0` 的未确认记录，且中心 ledger 健康  
**When** worker claim 并归集记录  
**Then** 中心事务为其分配从 1 开始、严格连续且不因事务回滚产生缺口的 `ledgerSequence`，保存完整最小化事实、`previousHash`、`entryHash`、hash profile 和来源版本，并原子更新 ledger head  
**And** 中心提交成功后，source relay 才以 claim fencing 条件将本地 outbox 确认为 `confirmed`；业务本地事实永不被删除或改写  
**And** 服务端授权、失败可见、审计、幂等和 `aggregateVersion` 冲突路径都有自动化或可复现验收。

### AC-1.4-CONTRACT：冻结 ledger、hash、重放与积压合同

**Given** 新的 `contracts/audit-ledger/` 受控契约族  
**When** contract checker 验证 ledger schema、hash profile、golden vectors、ingestion/replay policy、backlog policy、integrity finding schema、正反 fixture 与 SHA-256 lock  
**Then** 它们明确版本、必填/nullable、单位、边界比较符、genesis、字段顺序/规范化、双键幂等、错误码、恢复条件和隐私限制，并拒绝未知 major、未知字段、非法 hash/sequence、含敏感明文的 finding/alert 与 contract drift  
**And** 新 checker 同时消费并验证历史 `contracts/audit/**` v1 输入，但不修改其名为 `1.0.0` 的受控字节或历史 digest  
**And** 策略/公式发生语义变化时必须发布新版本并补向后兼容/拒绝测试，不得原位漂移。

### AC-1.4-SEQUENCE-AND-HASH：无缺口序号与确定性哈希

**Given** `ao_audit_ledger_head` 的 genesis 为 `ledgerSequence=0`、`entryHash=64` 个十六进制 `0`  
**When** 一个或多个 worker 并发追加、出现事务回滚或重试  
**Then** writer 在中心事务中以 `SELECT ... FOR UPDATE` 锁定单例 head，计算 `next=head+1`，插入 ledger 并更新 head；禁止用 PostgreSQL sequence/identity 充当无缺口序号  
**And** `entryHash = SHA-256(canonicalBytes(hashMaterial))`；hash material 至少绑定 domain tag、hash profile version、`ledgerSequence`、`previousHash`、`auditId`、`sourceEventId`、producer/event/fact schema 与 type、source/collected trusted time、`traceId`、`aggregateVersion` 和完整 payload fingerprint  
**And** canonical bytes 引用 `SCHOLARSENSE-CANONICAL-JSON-1.0.0` 的 UTF-8、键序、无多余空白和重复键拒绝语义，并以 audit-owned 固定字段模型与 golden vectors 固化；禁止依赖 Jackson/Map 偶然顺序、`jsonb::text`、数据库物理行字节或自由字符串拼接  
**And** verifier 使用常量时间字节比较，逐行重算 payload/hash，验证 sequence、`previousHash` 和最终 head。

### AC-1.4-IDEMPOTENCY：双键幂等与碰撞隔离

**Given** 中心 ledger 已存在一个已归集事实  
**When** 收到重复或冲突投递  
**Then** 同 `auditId + sourceEventId + payloadFingerprint` 的精确重试返回原 `ledgerSequence/entryHash`，不新增 ledger 行、不分配新序号，并只在独立 receipt/低基数指标中记录重复观察  
**And** 同 `auditId` 不同 `sourceEventId`、同 `sourceEventId` 不同 `auditId`、或任一相同标识关联不同 payload/版本/`aggregateVersion` 时，返回稳定 collision code，持久化 critical finding 与 alert outbox，不覆盖已有行、不确认 source outbox、也不分配新序号  
**And** 数据库唯一约束与 application 判定共同覆盖竞态，不能以捕获唯一键异常后误报普通成功替代内容一致性检查。

### AC-1.4-REPLAY：租约、fencing 与崩溃窗口可恢复

**Given** identity-access delivery role 只能操作自有 outbox 的 delivery metadata  
**When** relay 执行 `claim → center append → source confirm`  
**Then** claim 使用 `FOR UPDATE SKIP LOCKED`、`createdAt/eventId` due ordering 和递增 `attempts` 作为 fencing token，先提交源 claim；首版固定 `batchSize=100`、`claimLease=60s`，中心 append 在独立事务完成；source confirm/retry 更新必须匹配 `eventId + expected attempts`  
**And** worker 在中心 commit 后、source confirm 前崩溃时，租约到期重放必须得到同一中心结果并安全确认，不能产生第二条账本记录  
**And** 中心不可用、commit 失败或超时时，本地事实/outbox 仍保留并无限期重试；首版 retry cap 为 `min(60s, 2^(attempts-1)s)`，实际延迟使用该 cap 的 50%-100% jitter。alert transport 使用自身 outbox 重试；未知/非法 schema 或 collision 立即进入非重试 `failed/quarantine` 语义并产生 finding，不以无界热循环重试  
**And** claim 顺序仅为处理顺序，不能当作 producer 完整性序号；v1 继续不定义 `producerSequence`。

### AC-1.4-BACKLOG：版本化阈值与高风险失败关闭

**Given** 首版 `AUDIT-INGESTION-POLICY-1.0.0` 每 `15s` 以可信时间计算所有未确认状态（`pending/retrying/failed`）的数量与最老年龄，观测超过 `45s` 即视为陈旧  
**When** `oldestUnconfirmedAge >= 60s` 或 `unconfirmedCount >= 10,000`  
**Then** 可用性为 `degraded` 并发出去重 warning；低于边界不触发该状态  
**And** 当 `oldestUnconfirmedAge >= 300s`、`unconfirmedCount >= 50,000`、存在未处置的永久 `failed`/contract/collision finding、完整性 verifier 不健康，或状态测量不可得/超过政策规定的新鲜度时，可用性为 `blocked`  
**And** 任何高风险读写通过 `auditoperations.api.AuditAvailabilityPort` 检查；`blocked` 或依赖不可达时失败关闭并返回稳定错误码/`traceId`，不得返回敏感内容或提交业务变化  
**And** 只有在连续两次检查均满足 `oldestUnconfirmedAge < 60s`、`unconfirmedCount < 10,000`、无未处置的永久 failed/collision、chain/head 验证健康且观测新鲜时才恢复 `healthy`；处置只能记录独立 disposition 并保留原 outbox/finding，不可删改历史。相同 code/source-range 的 active alert 在 `300s` 窗口内去重并在状态恢复时产生一次 resolved 事件。以上阈值、`>=` 边界、观测周期/新鲜度、claim lease、退避、去重与恢复均写入该政策实例，任何改值必须新发政策版本和边界测试。

### AC-1.4-INTEGRITY-AND-ALERT：发现篡改且不泄露

**Given** 周期或人工 verifier 从受控 watermark/genesis 扫描 ledger  
**When** 发现序号缺口、`previousHash`/`entryHash` 不一致、head 漂移、幂等碰撞、合同拒绝或积压越界  
**Then** 持久化不可丢失的 finding 和独立 alert outbox，使用稳定 code：`AUDIT_LEDGER_SEQUENCE_GAP`、`AUDIT_LEDGER_PREVIOUS_HASH_MISMATCH`、`AUDIT_LEDGER_ENTRY_HASH_MISMATCH`、`AUDIT_LEDGER_HEAD_MISMATCH`、`AUDIT_INGESTION_DUPLICATE_CONFLICT`、`AUDIT_INGESTION_CONTRACT_REJECTED`、`AUDIT_INGESTION_BACKLOG`  
**And** finding/alert 仅含 code、severity、policy/hash version、sequence/source 范围、安全 digest、`traceId`、发生/发现时间和 runbook ref；不得含 actor/object/IP 明文、token/cookie、完整 payload、学生标识或证据正文  
**And** 精确传输重复只计数而不 page；内容碰撞和 chain/head 异常为 critical。低基数 metric label 不得放 `auditId/eventId/traceId`  
**And** alert transport 失败不能回滚一条正确的 ledger append，alert outbox 独立重试；chain/head 异常保持 `blocked`，不得自动改写/补写 ledger 伪造修复，恢复后必须全链验证并保留原 finding。

### AC-1.4-IMMUTABILITY：权限和所有权可执行

**Given** 全局下一号迁移 `V000003__audit-operations__immutable_ledger_v1.sql`  
**When** 在干净 PostgreSQL 18.4 和 V000001/V000002 升级路径应用迁移  
**Then** `audit_operations` 中的表均使用 `ao_` 前缀，至少包含 ledger、head、ingestion receipt、verification/finding 和 alert outbox，且无跨 schema FK  
**And** ledger writer 仅有必要的 `SELECT/INSERT`，对 ledger 无 `UPDATE/DELETE/TRUNCATE`；head、receipt、finding、alert 和 delivery 元数据通过各自最小列权限管理，verifier 为只读  
**And** `audit_operations` 不直接访问 identity 表；identity-owned relay 只依赖 `auditoperations.api` 的 transport-neutral ingress/availability contract  
**And** online role、delivery role、ledger writer、verifier 和未授权角色的越权负例均在真实 PostgreSQL 中失败。

### AC-1.4-RUNTIME：同制品 worker 能力受控加载

**Given** 当前同一后端制品支持 `web-api` 与无 HTTP 的 `worker` 角色  
**When** 受控部署启用 audit-ledger worker capability  
**Then** DataSource、可信时间、ingestion/backlog policy 与 collector/verifier 配置按环境引用加载，配置缺失、跨环境引用、静态伪造或过期时 fail fast 且不回显值  
**And** worker 不开放业务 HTTP，web-api 不启动 collector scheduler，identity web 行为和现有 remote-logout worker 行为不回归  
**And** 若 capability 写入 `deploy/base/roles.json`，同步升级契约 schema/checker/fixture；禁止只在 Java 中读取一个部署从未设置的“死开关”。

### AC-1.4-VERIFICATION：并发、故障、篡改和发布证据可复现

**Given** 候选实现改变 Java、SQL、contract 或发布受控字节  
**When** 申请进入 `review`  
**Then** unit/application/architecture 测试覆盖 canonical golden vectors（空/null、中文、非 BMP、键顺序、重复键、时间）、genesis、100+ 连续行、并发 worker、rollback 不耗号、精确重复、碰撞、stale claim、崩溃窗口、中心不可用、alert 不可用及积压所有精确边界/恢复滞回  
**And** 真实 PostgreSQL 18.4 测试用特权 test owner 注入 payload/`previousHash`/`entryHash`/head 修改和中间行删除，证明 verifier 分别发现；同时验证 writer/online/verifier 权限负例  
**And** 干净库与 V000001→V000002→V000003→V000004 升级路径、完整 `backend clean verify`、audit contract checker、`scripts/verify_core.sh`、`scripts/verify.sh` 和两次 clean-root 可复现构建通过  
**And** 新契约、脚本和迁移真实进入 source inventory、backend lock、SBOM、BuildManifest 和 ReleaseManifest；不得伪造 NTP、生产权限、WORM 或部署证据，也不得用 H2/内存替代 PostgreSQL 18.4。

## Dev Notes

### 不可协商的实施决策

- 中心顺序是“中心成功提交顺序”，不是 `occurredAt`、本地 surrogate 或 claim 顺序。PostgreSQL sequence/identity 会在 rollback 后留洞，不能满足本 Story 的严格连续语义。
- hash 链是 tamper-evident，不是单独的不可否认证明。online writer 与 verifier 权限分离可限制常规应用篡改，但数据库 owner/superuser 能重写整链；Story 1.5 必须以独立 WORM/归档回执补齐该威胁。
- `auditId` 与 `sourceEventId` 都是唯一业务键，payload fingerprint 是判断“精确重复还是碰撞”的必要证据；不得只凭一个键吞掉冲突。
- ledger 行只追加。投递、验证、告警状态永远位于独立技术表，不通过更新 ledger 表示确认、失败或修复。
- relay 属于 producer 模块；中心不扫描其他模块 schema。后续模块按同一 public API/producer-owned adapter 模式接入，不复制一个跨 schema 超级 repository。
- 不新增 runtime 依赖：JDK 25 的 `MessageDigest` 足以实现 SHA-256；使用现有 Spring JDBC/事务设施和 `UuidV7` 语义。
